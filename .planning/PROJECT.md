**v3.7.0** added the `Keeper` — a dedicated multi-replica recovery console on `BaseConsole.Core` that makes the autonomous execution loop **self-heal through transient L2 (Redis) outages without operator intervention.** Keeper subscribes to the `Fault<EntryStepDispatch>` / `Fault<ExecutionResult>` events the execution-path consumers publish on retry-budget exhaustion (pub/sub bind-to-all, never per-`{procId}_error`), double-unwraps the inner message + 6-id correlation + `H`, runs a bounded crash-survivable L2 health-probe loop (read `skp:data:{entryId}` + write-then-delete a scratch key, ack-after-loop so a mid-loop crash redelivers), pauses the affected workflow's cron through the single-replica orchestrator's Quartz scheduler (deterministic `TriggerKey` + `GetTriggerState` as the single source of truth — pause-state is deliberately NOT in L2, the resource being recovered), re-injects recovered work directly to its origin by type riding the receiver's surviving Phase-31 `flag[H]` (Keeper carries no dedup of its own), resumes on any successful recovery, and parks the genuinely unrecoverable in `keeper-dlq` for operator triage. A per-`H` attempt cap (atomic INCR+PEXPIRE-NX) bounds the recover→reinject cycle so a persistent fault converges to a single park instead of flooding the stack; transport-exhaustion across all consoles consolidates into one TTL'd `skp-dlq-1` wired once in `BaseConsole.Core` (two DLQs split by mechanism). ## Current Milestone: v4.0.0 — Processor Pre/In/Post-Process + Keeper Recovery Redesign

**Goal:** Replace the v3.x effect-first execution model with an explicit three-stage processor pipeline (Pre/In/Post-Process) and a proactive, Keeper-mediated L2-outage recovery model — retiring the `H`/dedup gate, content-addressing, the result manifest, and reactive `Fault<T>` recovery.

**Source of truth:** `docs/design/2026-06-08-processor-keeper-recovery-redesign.md` (LOCKED 2026-06-08). **Breaking change**; phases continue at **43**.

**Target features:**
- Explicit **Pre / In / Post-Process** pipeline in `BaseProcessor` (replaces the single `ProcessAsync` seam); author owns per-item outcomes + mints `executionId`; author-throwable `processing`/`failed`/`cancelled`.
- Reshaped contracts — 4 Keeper states (`UPDATE`/`REINJECT`/`INJECT`/`DELETE`) + reshaped `EntryStepDispatch`/`ExecutionResult`, **no `H`**.
- L2 lifecycle — GUID `entryId` (no TTL on the data key), composite backup key `corr:wf:proc:exec` (TTL 2 days, configurable), delete-on-consume (`finally` semantics).
- Keeper redesign — background **BIT health gate** + **global pause-all/resume-all** to all orchestrators (idempotent via Quartz `TriggerState`); gate-open-only recovery consumer applying the 4 states.
- Teardown of v3.x machinery — remove `H`/content-addressing/CAS dedup, the result manifest, reactive `Fault<T>` recovery; consolidate to a single `_DLQ1` (remove `keeper-dlq`).
- **At-least-once** — no dedup, duplicates tolerated; an absent/empty `L2[entryId]` read is `infra(READ)` → `REINJECT`, terminating at `_DLQ1` if the data is truly gone.

**Key context:** Recovery assumes a **transient** L2 outage (the recovery backup lives in L2 itself). This is the v4 breaking successor to v3.6.0's exactly-once-effect model — deliberately trading H-based exactly-once-effect for a simpler at-least-once + Keeper-replay model.

## Milestone History: v3.5.0 Processor Console — Self-Registration, Liveness & Execution Round-Trip

> **Status (2026-06-05):** v3.5.0 **shipped** 2026-06-02 and v3.6.0 **shipped** 2026-06-05; both complete. v3.7.0 (Keeper) is now the active milestone (see above). The detail below is retained as the v3.5.0 design record.

**Goal:** Stand up a reusable `BaseProcessor.Core` library + a first concrete `Processor.Sample` console (the processor-side mirror of `BaseConsole.Core`/`Orchestrator`) that self-identifies via an assembly-embedded SourceHash, self-registers its liveness into Redis L2, and runs the full live orchestrator→processor→orchestrator execution round-trip — with the actual transform isolated to one minimal `abstract ProcessAsync` seam.

**Target features:**
- **`BaseProcessor.Core` + `Processor.Sample`** — reusable Generic-Host scaffold built on `BaseConsole.Core` (soft-dep Redis, embedded health probes, metrics-only OTel, MassTransit/RabbitMQ, inherited correlation filters) + the first concrete console. Family convention: `Processor.<Purpose>`; `Processor.Sample` ships a POC dummy transform.
- **Assembly-embedded SourceHash** — MSBuild target (`BeforeTargets=CoreCompile`): SHA-256 (lowercase 64-hex, LF-normalized, per-file → folded) over `BaseProcessor.Core` + the concrete's `.cs`, emitted as `[assembly: AssemblyMetadata("SourceHash", …)]`, read at runtime by reflection. Re-runs on source change; excludes `BaseConsole.Core`/`Messaging.Contracts`.
- **WebApi bus responders** (relaxes the v3.4.0 publish-only firewall) — `GetProcessorBySourceHash` → `{ Id, InputSchemaId?, OutputSchemaId?, ConfigSchemaId? }` and `GetSchemaDefinition(schemaId)` → `{ Definition }`; the processor holds the `IRequestClient`s (first request/response on the console side).
- **Shared-contract extracts** to `Messaging.Contracts` — `ProcessorProjection` made public + relocated; the two new request/response contracts added; a new `L2ProjectionKeys.ExecutionData(entryId)` builder (`skp:data:{entryId:D}`).
- **Two-loop startup** — Loop A: bus-query identity by SourceHash, retry-on-failure → launch the liveness heartbeat worker → Loop B: bus-query the input + output definitions for each non-null schema Id, retry-on-failure → write to L2, exit.
- **Liveness heartbeat worker** — every `Interval`s write/refresh `skp:{processorId:D}` = `{ inputDefinition, outputDefinition, liveness{ timestamp, interval, status: "Healthy" } }` with `Ttl` expiry, written **only while Healthy** (identity + required definitions resolved). A starting/restarting replica doesn't write — to the orchestrator it's `absent`. Multi-replica safe & lock-free: the shared key is a blind whole-value `SET` of equivalent content (last-write-wins); execution-data keys are unique per result. The v3.4.0 `ProcessorLivenessValidator` is reused **unchanged** (presence+freshness ⟺ "≥1 replica healthy").
- **Execution round-trip** — consume `EntryStepDispatch` on `queue:{processorId:D}` → resolve input from `L2[data(entryId)]` (existence-checked) + validate vs `inputDefinition` (skip when empty) → `abstract ProcessAsync(inputData, config, ct)` where `config` = the dispatch `Payload` → per result: validate output vs `outputDefinition`, mint `entryId` + write `L2[data(newEntryId)]`, mint per-result `executionId` → send `ExecutionResult` one-by-one to `queue:orchestrator-result` → ack after all sends (empty list + no fault → ack only; Failed/Cancelled always sent).
- **Config** — two independent appsettings knobs: `Interval` (s; heartbeat delay, written as L2 `interval`) and `Ttl` (s; liveness key expiry); execution-data keys get their own configurable TTL.

**Key context / locked decisions:**
- The L2 reader half already shipped in v3.4.0 and expects this external self-registrar: `ProcessorLivenessValidator` (key `skp:{id:D}`, stale at `timestamp + interval×2`, 422 + RFC 7807) and the deliberately-empty `RedisProjectionWriter` (Phase 22 PROC-NOCREATE-01).
- The `Orchestrator` and the `EntryStepDispatch`/`ExecutionResult` wire contracts are reused **unchanged** — the orchestrator already forwards the carried-but-unused `EntryId`/`ExecutionId` end-to-end; this milestone gives them runtime meaning (`EntryId` = output-data L2 key, `ExecutionId` = per-result id).
- SourceHash is lowercase 64-hex to satisfy the existing DB validator (`^[a-f0-9]{64}$`); the Processor DB row is registered manually via CRUD using the hash read off the built binary; the processor retries identity until the row exists (boot-before-register).
- The virtual seam is `abstract Task<IReadOnlyList<ProcessResult>> ProcessAsync(string inputData, string config, CancellationToken ct)`; the framework owns all id-minting, validation, L2 read/write, and result sending so a new processor overrides only `ProcessAsync`.
- Input data is read only from L2 (`Payload` is config, never input data); the result contract stays outcome-only (no step-to-step data on the wire); processing relies on the orchestrator's liveness gate (no special dispatch-bind timing this milestone).

**Out of scope (deferred):** real (non-dummy) transform logic; step-to-step output-data forwarding on the wire; config re-validation in the processor; cleanup-on-read of execution-data keys; orchestrator-side entry-step L2 seeding.

## What This Is

A .NET 8 Web API platform that exposes CRUD over a workflow-engine data model — Schema, Processor, Step, Assignment, Workflow — built as a modular monolith on top of a reusable base library (`BaseApi.Core`) that future entities can plug into without rewriting infrastructure. The service ships logs to Elasticsearch and HTTP/runtime metrics to Prometheus via an OpenTelemetry Collector, runs against PostgreSQL via Docker Compose, and exposes RFC 7807 Problem Details on every error. Consumers (external Orchestrator + Scheduler) read Workflow entities to drive scheduled data-processing pipelines.

## Core Value

**A solid, observable, validated CRUD foundation that future workflow-platform features build on without rework.** If everything else fails, the base must reliably persist entities, validate inputs, emit telemetry, surface clean errors, and answer health probes. **Validated at v3.2.0 ship — base components were re-used 5× without rewrite for the 5 v1 entities; Phase 10 demonstrated that field-shape revisions land as bisect-friendly 5-commit sequences without infrastructure churn.**
