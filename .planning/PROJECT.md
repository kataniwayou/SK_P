- **Two-loop startup** тАФ Loop A: bus-query identity by SourceHash, retry-on-failure тЖТ launch the liveness heartbeat worker тЖТ Loop B: bus-query the input + output definitions for each non-null schema Id, retry-on-failure тЖТ write to L2, exit.
- **Liveness heartbeat worker** тАФ every `Interval`s write/refresh `skp:{processorId:D}` = `{ inputDefinition, outputDefinition, liveness{ timestamp, interval, status: "Healthy" } }` with `Ttl` expiry, written **only while Healthy** (identity + required definitions resolved). Multi-replica safe & lock-free: the shared key is a blind whole-value `SET` of equivalent content (last-write-wins); execution-data keys are unique per result. - **Execution round-trip** тАФ consume `EntryStepDispatch` on `queue:{processorId:D}` тЖТ resolve input from `L2[data(entryId)]` (existence-checked) + validate vs `inputDefinition` (skip when empty) тЖТ `abstract ProcessAsync(inputData, config, ct)` where `config` = the dispatch `Payload` тЖТ per result: validate output vs `outputDefinition`, mint `entryId` + write `L2[data(newEntryId)]`, mint per-result `executionId` тЖТ send `ExecutionResult` one-by-one to `queue:orchestrator-result` тЖТ ack after all sends (empty list + no fault тЖТ ack only; Failed/Cancelled always sent).
- **Config** тАФ two independent appsettings knobs: `Interval` (s; heartbeat delay, written as L2 `interval`) and `Ttl` (s; liveness key expiry); execution-data keys get their own configurable TTL.

**Key context / locked decisions:**
- The L2 reader half already shipped in v3.4.0 and expects this external self-registrar: `ProcessorLivenessValidator` (key `skp:{id:D}`, stale at `timestamp + interval├Ч2`, 422 + RFC 7807) and the deliberately-empty `RedisProjectionWriter` (Phase 22 PROC-NOCREATE-01).

- Input data is read only from L2 (`Payload` is config, never input data); the result contract stays outcome-only (no step-to-step data on the wire); processing relies on the orchestrator's liveness gate (no special dispatch-bind timing this milestone).

**Out of scope (deferred):** real (non-dummy) transform logic; step-to-step output-data forwarding on the wire; config re-validation in the processor; cleanup-on-read of execution-data keys; orchestrator-side entry-step L2 seeding.

## What This Is

A .NET 8 Web API platform that exposes CRUD over a workflow-engine data model тАФ Schema, Processor, Step, Assignment, Workflow тАФ built as a modular monolith on top of a reusable base library (`BaseApi.Core`) that future entities can plug into without rewriting infrastructure.
