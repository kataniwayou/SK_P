<objective>
Make the keeper-recovery endpoint symmetric with the processor/orchestrator execution-path endpoints. The keeper consumers ALREADY wrap every L2 op + Send in the in-code `RetryLoop` via `RecoveryConsumerBase.Guard`, which re-throws on exhaustion. The ONLY divergence is the bus-level config in `RecoveryEndpointBinder`: a policy-conditional `UseMessageRetry` block plus a `ConfigureError` block that routes exhausted ops to `skp-dlq-1`. Remove both so a Guard-exhaust throw falls through to RabbitMQ nack-requeue (broker redelivery) — no `_error`, no dead-letter — exactly like `ProcessorStartupOrchestrator`'s dispatch endpoint and the orchestrator result endpoint.

This is a deletion/refactor, not new logic. It also sweeps the now-dead `ExhaustionPolicy` surface (enum, field, base property, ctor params on the base + 3 subclasses, appsettings key) and re-points the two tests that asserted the removed behaviors.

Purpose: kill the last keeper-local divergence from the A18 "no bus retry / no error transport on the execution path" end-state.
Output: a symmetric keeper-recovery endpoint, a clean 0-warning build, and a hermetic Keeper suite that proves the new nack-requeue/no-dead-letter posture.
</objective>
