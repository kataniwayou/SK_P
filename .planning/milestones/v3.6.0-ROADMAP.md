# Milestone v3.6.0: Idempotent Execution — Exactly-Once-Effect Round-Trip

**Status:** ✅ SHIPPED 2026-06-05

**Total Plans:** 9 active (31: 6/6 · 31.1: 1/1 · 32: 5/6 superseded · 32.1: 2/2)

## Overview

The orchestrator↔processor execution round-trip becomes **exactly-once-effect**: every duplicate inbound message — from configurable `Immediate(N)` retry, broker redelivery, publish-confirm ambiguity, the orchestrator's own re-dispatch, or a fan-in/merge re-dispatch — reproduces the same deterministic identity `H = SHA-256(correlationId, workflowId, stepId, processorId, EntryId)` and is collapsed at the receiving node, so each step's downstream effect happens exactly once with **no lost branch**, and the processor transform (`ProcessAsync`) needs no idempotency logic. Dedup is **effect-first** (the downstream effect is produced before the atomic `flag[H]` `Pending → Ack` CAS), so the irreducible residual — a crash between send and flip — is a downstream-collapsed duplicate, never a loss. L2 data is content-addressed at two levels (result blobs + manifest); merge steps are per-edge (distinguished by input `EntryId`, identical-input collapses); the orchestrator fans the manifest out N×M. The milestone also flipped the hard-coded `Immediate(3)` to a configurable retry budget and standardized content-addressed `data`/`flag` keys to `prefix + 64-hex`.

A late course-correction (Phase 32 → 32.1) **reverted** the originally-planned "cancelled circuit-breaker": after building it, the two-level breaker was judged not worth its surface area and was net-deleted in favor of plain dead-lettering to `_error` on retry-budget exhaustion. The Phase-31 idempotency gates survived the revert intact (re-confirmed 5/5 by the integration checker).

## Milestone Summary

**Decimal Phases:**

- Phase 31.1: Close-Gate Redis Net-Zero (inserted after Phase 31 — gap closure for the redis net-zero leak surfaced by the live close gate)
- Phase 32.1: Dead-Letter on Exhaustion (inserted after Phase 32 — reverts the cancelled circuit-breaker to plain dead-lettering)

**Key Decisions:**

- Receiver dedup, not producer detection (forced by at-least-once delivery + one-directional confirmation loss) — ✓ Good
- Effect-first `flag[H]` CAS (effect before `Pending → Ack` flip) → exactly-once-EFFECT, residual is a collapsed duplicate, never a loss — ✓ Good
- `H = SHA-256(correlationId, workflowId, stepId, processorId, EntryId)`, executionId excluded; single canonical `MessageIdentity` hash class — ✓ Good
- Content-collapse merge (identical input → same `H` → one execution); `predecessorStepId` NOT in `H` — ✓ Good
- Content-addressed two-level L2 (`data[hash(blob)]` + manifest at `data[hash(manifest)]`); `prefix + 64-hex` keys — ✓ Good
- Configurable retry budget (`Immediate(N)` default, back-off available) replacing hard-coded `Immediate(3)` — ✓ Good
- **Cancelled circuit-breaker REVERTED** (Phase 32 → 32.1): plain dead-lettering on exhaustion; "stop the job on exhaustion" off the table — ✓ Good (conscious scope removal)

**Issues Resolved:**

- Fan-in/merge double-execution (`StepB4` ×2, ES-confirmed) — collapsed via per-edge input `EntryId`
- Retry/redelivery duplicate downstream execution — collapsed via effect-first `flag[H]` dedup at both hops
- Close-gate redis net-zero leak (per-fire `skp:flag:{H}` name churn from self-rescheduling cron E2E) — Phase 31.1
- Breaker over-engineering (no-TTL marker + manual resume + best-effort future-fire stop) — net-deleted in Phase 32.1

**Issues Deferred / Accepted Consequences:**

- A persistently infra-faulting workflow is NOT stopped — its Quartz job keeps firing and each fire dead-letters one message per cron period until an operator intervenes (`POST /orchestration/stop` or `_error` triage). Explicit trade for dropping the breaker.
- Transactional outbox (atomic flag+send to close even the collapsible-duplicate window) — deliberately deferred; effect-first + downstream dedup chosen instead.
- `ProcessAsync` external-side-effect idempotency — out of framework scope (exactly-once-EFFECT, not exactly-once-EXECUTION; `ProcessAsync` must be pure or independently idempotent).
