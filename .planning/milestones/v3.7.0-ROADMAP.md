# Milestone v3.7.0: Keeper — L2-Outage Dead-Letter Recovery & Workflow Pause/Resume

**Status:** ✅ SHIPPED 2026-06-07
**Phases:** 33-42 (10 phases)
**Total Plans:** 32

## Overview

A dedicated multi-replica `Keeper` console that makes the autonomous execution loop (cron fire → dispatch → process → result → fan-out) **self-heal through transient L2 (Redis) outages without operator intervention.** Keeper reacts to the `Fault<EntryStepDispatch>` / `Fault<ExecutionResult>` events that the execution-path consumers publish on retry-budget exhaustion (pub/sub, bind-to-all, never per-`{procId}_error`), probes L2 health on a bounded crash-survivable loop, pauses the affected workflow's cron (via the single-replica orchestrator's Quartz state) so the outage stops spreading, re-injects recovered work to its origin by type (riding the receiver's surviving Phase-31 `flag[H]` idempotency — Keeper carries no dedup of its own), resumes when a recovery succeeds, and parks the genuinely unrecoverable in a single `keeper-dlq` for operator triage. Keeper is the automated operator for v3.6.0's accepted "an infra-faulting workflow keeps dead-lettering until an operator intervenes" gap. Operator commands (Start/Stop) are out of scope — a failed Start/Stop is simply re-issued.

The milestone delivered its definition of done **live-proven** (Phase-39 close gate: 3× consecutive GREEN at 500 RealStack facts each, triple-SHA net-zero held). A second audit (2026-06-07) added three gap-closure phases (40 recovery hardening, 41 pause/resume diagnostics, 42 docs reconciliation) over the original 33-39 build.

## Milestone Summary

**Gap-Closure Phases (no decimal insertions this milestone):**

- Phase 40: Keeper Recovery Hardening — from the 2026-06-07 audit (attempt cap, deterministic drain, shared handler)
- Phase 41: Orchestrator Pause/Resume Diagnostics — closes 37-REVIEW WR-01/WR-02
- Phase 42: v3.7.0 Docs & Traceability Reconciliation — doc drift + missing 39-VERIFICATION.md

**Key Decisions:**

- Keeper recovers off `Fault<T>` **pub/sub** (bind-to-all), never per-`{procId}_error` queues — one consumer covers all producing replicas (D-10 `_error` retained as TTL'd forensic copy).
- Re-inject **directly to origin by type** (`queue:{procId:D}` / `queue:orchestrator-result`), no orchestrator round-trip; idempotency rides the receiver's surviving Phase-31 `flag[H]` — Keeper holds no dedup of its own.
- Pause-state lives in the **single-replica orchestrator's Quartz scheduler** (`GetTriggerState` via deterministic `TriggerKey`), NOT L2 — because L2 is the resource being recovered; no separate L1 state field (D-05/D-08).
- Two DLQs **split by mechanism**: `skp-dlq-1` (Immediate(N) transport exhaustion, consolidated across all consoles, 7d TTL, wired once in `BaseConsole.Core`) + `keeper-dlq` (probe/cap give-up park, depth = primary alert).
- Recover→reinject **bounded by a per-`H` attempt cap** (Phase 40) so a persistent fault converges to a single park, not an unbounded reinject flood.
- Metrics `service_name = {name}_{version}` with the **processor's name/version DB-sourced** via a MeterProvider swap; logs `service.name` stays bare.

**Issues Resolved:**

- Unbounded recover→reinject loop for persistent faults (~67 cyc/s/replica) → per-`H` attempt cap (Phase 40 KHARD-01).
- `keeper-dlq` depth==0 close-gate race on late give-up parks → poll-until-stably-empty drain (Phase 40 KHARD-02).
- Near-total duplication across the two Keeper fault consumers → shared `KeeperRecoveryHandler` (Phase 40 KHARD-03 / IN-01).
- 37-REVIEW WR-01 (silent dropped-Resume) + WR-02 (RescheduleAsync throws on purged job) → Phase 41 diagnostics.
- REQUIREMENTS/ROADMAP doc drift + missing 39-VERIFICATION.md → Phase 42 reconciliation.
- A load-bearing 37-02 self-reschedule regression (`ScheduleJob`→`RescheduleJob`) caught + fixed during execution (`571498f`).
