<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| recovery queue → Keeper consumer | Recovery envelopes from the trusted processor cross into L2 ops + bus sends. |
| Keeper consumer → Redis (L2) | Each state body issues L2 read/write/delete; faults must surface, not be swallowed. |
| Keeper consumer → bus (re-dispatch / result injection) | REINJECT re-sends to a processor queue; INJECT sends to the orchestrator result queue. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-46-03 | Denial of Service | Poison message looping on keeper-recovery | mitigate | Every L2 op + Send is bounded by RetryLoop(RetryLimit); on exhaustion the body re-throws → endpoint UseMessageRetry (Plan 03) → skp-dlq-1. The gate-wait is bounded by GateWaitSeconds (linked CTS) so no indefinite un-acked hold. |
| T-46-04 | Denial of Service | Indefinite gate-closed hold | mitigate | The linked CTS bounds the WaitForOpenAsync to GateWaitSeconds (default 300s, well under the 30-min consumer_timeout); on bound a transient RecoveryGateTimeoutException routes through the retry policy. |
| T-46-05 | Tampering / correctness | Composite key collision via partition hash (Plan 03) | accept | Murmur3 is a non-crypto distribution hash; collisions only co-schedule unrelated execs (ordering optimization). Identity is the full 4-tuple L2 key, unaffected. |
| T-46-06 | (none) | Duplicate recovery effects (at-least-once) | accept | Explicitly tolerated by design (REQUIREMENTS.md:54,72); INJECT (write-then-delete) / REINJECT (re-dispatch) are idempotent-enough. No dedup key introduced. |
</threat_model>
