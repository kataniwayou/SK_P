<objective>
Add `ProcessorId` enrichment and minted-id scoping on the processor side: (1) a custom OTel `BaseProcessor<LogRecord>` (`ProcessorIdLogEnricher`) that appends `ProcessorId` from the singleton `IProcessorContext.Id` to EVERY processor log (startup, heartbeat, consume) — null-safe (emits nothing before identity resolves), registered ONLY on the processor's logger provider; and (2) a nested `BeginScope` in `EntryStepDispatchConsumer`'s Completed path carrying the per-result minted `ExecutionId` and output `EntryId` on the write/send log lines (MEL inner-overrides-outer, so those lines report the minted ids rather than the inbound `Guid.Empty` the filter skipped).

Purpose: `ProcessorId` is null until identity resolves, so a scope can't cover pre-identity logs — only a logger-provider processor reaching the process-wide singleton can (D-04). The minted `ExecutionId`/`EntryId` do not exist until mid-consume, so only the consumer body can scope them (D-05). Both surface at ES `attributes.ProcessorId` / `attributes.ExecutionId` / `attributes.EntryId` via the unchanged OTel bridge (LOG-01). The enricher is the ONLY genuinely new mechanism in this phase; it MUST live in `BaseProcessor.Core` (it depends on `IProcessorContext`), NEVER in the shared `BaseConsole.Core` observability extension (L3 — that would invert layering and break the orchestrator, which has no `IProcessorContext`).

Output: the enricher, its processor-side registration, the nested scope, and two hermetic tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| processor identity (singleton) → log telemetry | `IProcessorContext.Id` (server-resolved Guid, null pre-identity) crosses into every LogRecord via the enricher. |
| consume body (minted ids) → log telemetry | The consumer's minted `ExecutionId`/`EntryId` (server-minted via `NewId.NextGuid()`) cross into the nested scope on the write/send lines. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-05 | Information Disclosure (sensitive data in log attributes) | ProcessorId / ExecutionId / EntryId attributes | accept | All three are server-minted/resolved GUIDs — no PII/secret. `ProcessorId` is null-safe (nothing emitted before identity, never `Guid.Empty`). No sensitive data newly placed into attributes. |
| T-29-06 | Tampering / Repudiation (log injection) | nested-scope values + enricher value | accept | Values are typed `Guid` placed only as scope/attribute VALUES under fixed keys (T-18-04), never interpolated into a template; not user-controlled. Injection surface nil. |
| T-29-07 | Denial of Service (enricher throws/blocks on the hot logging path) | ProcessorIdLogEnricher.OnEnd | mitigate | OnEnd is non-blocking, allocation-light, and cannot throw on the null path (`if (context.Id is not { } id) return;`); reads a volatile-ish singleton property only. Verified by the null-case test (Case B asserts no exception). Per OTel contract, OnEnd must not throw — the null-guard guarantees it. |

No `high` (block-worthy) threats: the data crossing both boundaries are server-side opaque GUIDs placed as structured values; the enricher reads one singleton property and cannot throw. Reasoning: no user-controlled input, no new sink, and the only reliability risk (a throwing/blocking enricher) is mitigated by the null-guard and verified by the null-case test.
