<decision>
**LOCKED — D-15 / Pitfall 5: an L2 OUTPUT-WRITE fault is INFRA → it THROWS** (whole dispatch retries via `Immediate(3)`; never lose output). This is grounded in `WorkflowLifecycle.IsInfra` classifying `RedisConnectionException`/`RedisTimeoutException`/`RedisException` as infra-that-propagates. Do NOT wrap the output `StringSetAsync` in a catch — let it propagate. By contrast, output-VALIDATION failure (the schema check) stays a business `Failed`. The input-read `StringGetAsync` also propagates Redis faults (infra). Accepted at-least-once tradeoff: throwing mid-batch re-sends already-sent results on retry — the orchestrator's `ResultConsumer` is L1-idempotent, so this is safe (document it in the SUMMARY).
</decision>

<decision>
**LOCKED — open question 2 (no-input source processor):** when `InputDefinition` is null/whitespace AND `EntryId` is `Guid.Empty`, do NOT read L2 and pass `inputData = ""` to `ProcessAsync` (a source processor legitimately has no input). When `InputDefinition` is null/whitespace but `EntryId` is non-empty, still read L2 (use the value if present, else `""`) but skip validation. When `InputDefinition` is non-empty and `EntryId` is empty/missing-from-L2 → single `Failed` before `ProcessAsync` (D-07).
</decision>

<decision>
**LOCKED — send provider:** inject `ISendEndpointProvider` into the consumer ctor (mirrors `StepDispatcher`); resolve `queue:orchestrator-result` once per consume and `Send` each result one-by-one. (Using `context` as the provider is equivalent — but ctor injection matches `StepDispatcher` and is easier to fake.)
</decision>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| orchestrator -> queue:{id:D} (dispatch body) | An untrusted `EntryStepDispatch` (`Payload`, `EntryId`) crosses into the consumer. |
| L2[data(entryId)] -> ProcessAsync / schema eval | Untrusted input data read from Redis fed to schema validation + the concrete transform. |
| ProcessAsync output -> L2[data(newEntryId)] | Transform output written to the shared L2 keyspace. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-27-04 | Tampering / Information Disclosure (SSRF) | input/output schema evaluation in the consumer | mitigate | Validation goes through `ProcessorJsonSchemaValidator` (Plan 01) whose static ctor installs the no-op global `$ref` fetcher; the consumer never builds its own `EvaluationOptions`. |
