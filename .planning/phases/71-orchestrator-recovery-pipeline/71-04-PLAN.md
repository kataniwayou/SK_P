## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| keeper-recovery queue → consumers | Inbound OrchestratorInject/OrchestratorReinject cross from the broker into the keeper consumers. Producer is the orchestrator pipeline (internal), not an external client. |
| consumers → Redis / orchestrator-result | The INJECT consumer writes a copied data key; the REINJECT consumer re-injects an IStepResult to orchestrator-result. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-71-09 | Elevation (over-deletion) | OrchestratorInjectConsumer / OrchestratorReinjectConsumer | mitigate | Neither consumer calls KeyDeleteAsync (grep-asserted, Task 1) and the behavioral D-09 facts prove DidNotReceive on BOTH KeyDeleteAsync overloads with a positive co-assertion (so a silent no-op cannot pass). KeeperDelete/DeleteConsumer remains the only deleter. |
| T-71-10 | Tampering / Spoofing | Routing by outcome discriminator in OrchestratorReinjectConsumer | mitigate | The only status branch is the bounded `m.Outcome switch` with an exhaustive default (`_ => StepFailed`); an unknown/invalid outcome degrades to a safe StepFailed re-inject, never an exception or a mis-typed result. No reflection / no polymorphic blob deserialization on the wire. |
| T-71-11 | Denial of Service | Static+connect endpoint collision on keeper-recovery | mitigate | Both consumers registered with ExcludeFromConfigureEndpoints() (Pitfall 6); the binder owns the endpoint. No new queue/endpoint surface; same health gate / partitioner / exhaustion posture reused verbatim. |
| T-71-12 | Information Disclosure | ErrorMessage / CancellationMessage carried on OrchestratorReinject | accept | Same diagnostic strings already on orchestrator-result via StepFailed/StepCancelled; internal broker only, no external consumer — no new exposure. |
