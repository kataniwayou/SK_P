## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dispatch message → framework deserialize | The dispatch `payload` (string) crosses into `BaseProcessor<TConfig>.ExecuteAsync` and is fed to `JsonSerializer.Deserialize<TConfig>`. The payload is orchestration-admitted (Gate B schema-validated upstream in the WebAPI) but is the untrusted-shaped input at THIS seam. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-56-01 | Denial of Service / Tampering | `BaseProcessor<TConfig>.ExecuteAsync` deserialize of a malformed/non-`TConfig` `payload` | mitigate | Do NOT catch the `JsonException` locally and do NOT return a default config (no silent default). Let it propagate to the existing pipeline catch-all (`ProcessorPipeline.cs:241`) which maps it to exactly one business `StepFailed`, logs it, and aborts the batch deterministically — no crash that escapes the pipeline, no un-acked redelivery, no infra/Keeper mis-route (D-03). Proven by the deser-failure fact added in Plan 02. |
| T-56-02 | Tampering / Information Disclosure | Empty/whitespace/absent `payload` | mitigate | `string.IsNullOrWhiteSpace(payload)` guard short-circuits to a `null` config BEFORE deserialize (D-04) — `Deserialize` is never called on an empty string (which would otherwise throw), so the config-less path is deterministic and the author owns null-handling. |
| T-56-03 | Tampering | Extra / differently-cased JSON properties in the `payload` | accept | The shared `JsonSerializerOptions` is deliberately forgiving (case-insensitive, unknown-member IGNORED) per D-05 — an admitted payload always deserializes. Residual/accepted risk: a payload carrying unexpected extra keys deserializes silently (keys ignored). This is by design — strictness is Gate B's job (upstream schema validation, unchanged) and Gate A's job (Phase 57 config-type coverage). Adding `JsonUnmappedMemberHandling.Disallow` here is explicitly OUT of scope (deferred to Phase 57). No PII, no auth, no new persistence/network surface in this phase. |
</threat_model>
