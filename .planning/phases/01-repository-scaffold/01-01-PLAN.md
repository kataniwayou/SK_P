**Severity behavior note (W-03 — document `:warning` composition with EnforceCodeStyleInBuild + TreatWarningsAsErrors)**

Every rule in this .editorconfig whose severity is `:warning` (e.g., `csharp_style_namespace_declarations = file_scoped:warning`, `csharp_using_directive_placement = outside_namespace:warning`, and all four `dotnet_naming_rule.*.severity = warning` lines) is REPORTED AS A WARNING by the analyzer. By itself, that is informational — but it composes with two other settings established by Plan 01 Task 2 (`Directory.Build.props`):

- `<EnforceCodeStyleInBuild>true</EnforceCodeStyleInBuild>` — makes the analyzer run during `dotnet build`, not only inside an IDE. Without this, `:warning` rules would be silent at compile time.
- `<TreatWarningsAsErrors>true</TreatWarningsAsErrors>` — promotes ALL warnings (including these style/naming warnings) to errors during build.

The net effect: every `:warning` rule in this file is BUILD-FATAL in SK_P. A compile that violates any `:warning` rule FAILS — it does not produce a warning the developer can ignore. This is intentional per D-02.

Concrete examples of code that WILL FAIL THE BUILD in later phases because of `:warning` rules composed with TreatWarningsAsErrors:

- `namespace BaseApi.Core.Entities { public class Foo { } }` — block-scoped namespace, violates `csharp_style_namespace_declarations = file_scoped:warning`. Correct form: `namespace BaseApi.Core.Entities;` followed by the class declaration.
- `namespace BaseApi.Core { using System; public class Foo { } }` — `using` inside namespace, violates `csharp_using_directive_placement = outside_namespace:warning`. Correct form: `using System;` ABOVE the namespace declaration.
- `public class foo { }` — lowercase type name, violates `dotnet_naming_rule.types_should_be_pascal_case` (severity `warning`). Correct form: `public class Foo { }`.
- `public class MyService { private string name; }` — private field without underscore prefix, violates `dotnet_naming_rule.private_fields_should_be_camel_case_with_underscore` (severity `warning`). Correct form: `private string _name;`.
- `public interface MyService { }` — interface without `I` prefix, violates `dotnet_naming_rule.interface_should_be_begins_with_i` (severity `warning`). Correct form: `public interface IMyService { }`.
- `if (cond) doThing();` — single-line if without braces. This baseline has `csharp_prefer_braces = true:silent`, so it is NOT build-fatal here. Listed only to illustrate the asymmetry: `:silent` rules surface in the IDE but never break the build.

Future-phase planners and executors: do NOT add code that violates any rule whose severity in this file is `:warning` (or stricter). Either fix the code, or — if there is a legitimate reason — propose tuning the rule's severity DOWN to `:suggestion` or `:silent` in this file as a deliberate plan-level decision, NOT via inline `#pragma warning disable`. The composition with TreatWarningsAsErrors is the mechanism by which Phase 1 SC#1 ("zero warnings") becomes self-enforcing.

Rules with severity `:silent` or `:suggestion` in this file do NOT become build-fatal — they exist as IDE hints only. The asymmetry is intentional: this baseline picks which rules are non-negotiable (named-style + namespace style + using placement) versus which are stylistic preferences IDEs surface (var usage, expression-bodied members, etc.).
