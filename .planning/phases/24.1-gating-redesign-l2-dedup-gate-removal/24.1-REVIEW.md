# Phase 24.1: Code Review Report

**Reviewed:** 2026-06-01
**Depth:** standard
**Files Reviewed:** 18
**Status:** issues_found

## Summary

Reviewed the gap-closure phase 24.1 implementation: the WebApi-side L2-existence dedup with
parent-index compensation (`OrchestrationService`), the atomic discover-then-delete Stop cleanup
(`RedisL2Cleanup`), removal of the boot gate / scheduled redelivery / delayed-message-scheduler plugin
dependency across the three orchestrator consumers and their definitions plus `Program.cs`, and the
terminal-step `NextStepIds` null-guard in `StepAdvancement`.

Overall the implementation is coherent and well-documented, and the design contracts (first-win
dedup, probe-not-delete Stop, L1-only graceful result path, atomic delete batch, R2 compensation) are
faithfully realized and backed by targeted tests. No critical bugs or security issues were found. The
single-replica WebApi assumption and the accepted probe→write TOCTOU window are explicitly documented
and out of scope per the phase brief.

Two warnings concern an asymmetry in the Start-path R2 compensation versus the gone-stale doc/op-name
labels, and a partial-failure window inside the cleanup's own multi-call sequence that the R2
compensation does not fully cover. The info items are minor consistency / clarity observations.

## Warnings

### WR-01: Start-path R2 compensation can over-SREM and remove a concurrently/previously valid parent-index entry

**File:** `src/BaseApi.Service/Features/Orchestration/OrchestrationService.cs:213-231`
**Issue:** On a Start `UpsertAsync` fault, the catch issues a compensating
`SetRemoveAsync(ParentIndex, workflowId)`. This is correct for the first-win path where the root was
just confirmed absent (the id is being newly written, so removing it on failure is right). However the
pre-clean step at line 170 (`_cleanup.StopCleanupAsync`) has *already* unconditionally SREM'd this id
from the parent index (RedisL2Cleanup.cs:45) before the Upsert ran. So on the Upsert fault path the
SREM compensation is redundant (the id is already absent from the index) — harmless functionally, but
it means the "compensation" is a no-op in the normal flow and the real index-consistency guarantee
actually comes from the pre-clean SREM, not the catch block. The comment at lines 207-212 attributes
the consistency guarantee to this SREM, which is misleading. This is a correctness-of-reasoning issue
rather than a runtime defect: the index ends consistent either way, but a future maintainer relying on
the documented invariant ("SREM this id ... so the index never retains an id whose L2 write did not
land") may not realize the actual enforcing operation is the pre-clean, and could remove the pre-clean
SREM believing the catch covers it.
**Fix:** Clarify the comment to state the parent-index removal is already guaranteed by the pre-clean
`StopCleanupAsync` (which SREMs before any write), and that the catch-block SREM is a defensive
belt-and-suspenders for any future path that writes without a preceding pre-clean. Alternatively, drop
the catch-block SREM and rely solely on the atomic Upsert batch + pre-clean, documenting that choice —
fewer Redis round-trips on the fault path and one less place to reason about.

### WR-02: `RedisL2Cleanup.StopCleanupAsync` is not atomic across its full sequence; a fault between the SREM and the delete batch leaves the parent index inconsistent with a still-present root

**File:** `src/BaseApi.Service/Features/Orchestration/Projection/RedisL2Cleanup.cs:45-95`
**Issue:** The routine performs, in order: (1) `SetRemoveAsync(ParentIndex, id)` (line 45), (2) one or
more `StringGetAsync` reads to discover the step set (lines 49-78), (3) one atomic `CreateBatch` delete
of root + steps (lines 91-95). Only step (3) is atomic. If a Redis fault occurs during the discovery
GETs (step 2) — after the SREM in step (1) succeeded but before the delete batch — the parent index no
longer contains the id, yet the root + step keys are still fully present in L2. The `StopAsync` caller
catches the `RedisException` and SADDs the id back (OrchestrationService.cs:341-349), which does
compensate this specific window. But the Start-path caller's pre-clean (OrchestrationService.cs:168-176)
catches and rethrows WITHOUT re-adding — acceptable there only because the Start path is about to
fail the whole request anyway. The subtler issue: the doc comment on the cleanup (lines 80-90) and on
`StopAsync` describes the delete as "ONE atomic batch ... all-or-nothing," which is true for the delete
itself but understates that the SREM is a *separate, earlier, non-batched* op. The R4 reachability
invariant argument (lines 80-85) justifies that no orphan per-step key can exist, but it does not
address the SREM-then-fault-before-delete window.
**Fix:** Either (a) move the parent-index SREM *into* the delete batch so SREM + root-delete + step-delete
are genuinely one atomic unit (the comment at line 44 explicitly forbids this today to keep the SREM
idempotent above the absent-root early return — but the early-return SREM and the batch SREM could both
be retained only if guarded against double-SREM, or the early-return case could SREM standalone while
the present-root case SREMs inside the batch); or (b) keep the current ordering but tighten the doc
comment to state explicitly that only the final delete is atomic and that parent-index/root consistency
across the SREM→delete window relies on the caller's compensation (StopAsync SADD), so the invariant is
caller-enforced, not self-contained. Given the locked single-replica assumption and the StopAsync SADD
compensation, runtime risk is low — this is primarily a documentation/contract-tightening fix.

## Info

### IN-01: Stale doc reference to the Phase-22 cleanup loop ("traverse-and-delete of the per-step keys") in test comment

**File:** `tests/BaseApi.Tests/Features/Orchestration/StopGateFacts.cs:230-233`
**Issue:** The comment describes the post-delete cleanup as "the root KeyDeleteAsync returned true →
tolerant traverse-and-delete of the per-step keys," which is the superseded Phase-22 delete-root-first
ordering. Under 24.1 (R1/R3 probe-not-delete) the caller PROBES with `KeyExistsAsync` and the cleanup
reads the still-present root to discover step keys, then deletes root + steps in one batch. The test
itself is correct (it injects a throwing cleanup and asserts redisOp="KeyDeleteAsync"), only the
explanatory comment lags the new ordering.
**Fix:** Update the comment to reflect probe-then-cleanup: "the per-id KeyExistsAsync probe returned
true → StopCleanupAsync reads the root, BFS-collects step keys, and deletes root+steps in one batch."

### IN-02: `redisOp` tag value "UpsertAsync" applied to the KeyExistsAsync probe and pre-clean ops obscures the actual faulting operation

**File:** `src/BaseApi.Service/Features/Orchestration/OrchestrationService.cs:146-152, 168-176, 193-201`
**Issue:** On the Start path, a `RedisException` from the existence `KeyExistsAsync` probe (line 146),
the pre-clean `StopCleanupAsync` (line 170), and the processor-liveness GETs (line 195) are all tagged
`Data["redisOp"] = "UpsertAsync"`. This is a deliberate, documented choice (OBSERV-REDIS-03: a single
stable op name for the whole Start L2-write path so the 500 body is caller-stable) and the
corresponding Stop-path tag is uniformly "KeyDeleteAsync". It is intentional and tested, but worth
noting that the surfaced op name does not identify which Redis call actually faulted, which can
complicate operational debugging from the 500 body alone (logs/traces would still carry the real op).
**Fix:** No code change required if the stable-op-name contract is intended. If operational diagnosis
matters, consider keeping the stable caller-facing `redisOp` but additionally logging the true faulting
op name at the catch site (it is not logged today on these paths).

### IN-03: `using System.Linq;` explicit import is redundant under the project's likely ImplicitUsings

**File:** `src/Orchestrator/Dispatch/StepAdvancement.cs:1`
**Issue:** `using System.Linq;` is explicitly imported. Sibling files in the same project
(e.g. ResultConsumer.cs, the consumers) do not import it explicitly and rely on implicit usings /
existing references, suggesting `ImplicitUsings` is enabled. The explicit import is harmless but
inconsistent with the surrounding files. (`Enumerable.Empty<Guid>()` at line 40 needs System.Linq,
which implicit usings already provide.)
**Fix:** Remove the explicit `using System.Linq;` for consistency, or leave as-is — purely stylistic.

### IN-04: `Program.cs` comment block describes the fan-out endpoint as "orchestrator-{instanceId}" but the boot-gate-removal comment is the only 24.1 marker; the instanceId fallback uses a fresh GUID per process

**File:** `src/Orchestrator/Program.cs:23-42`
**Issue:** Two minor observations. (1) The `instanceId` fallback `Guid.NewGuid().ToString("N")` (line 27)
means an unconfigured replica gets a new fan-out queue name on every restart; under the auto-delete
`Temporary = true` endpoints this is intentional and correct, but it does mean a crash-restart abandons
the prior temporary queue until the broker reaps it — acceptable and consistent with prior phases.
(2) The 24.1 removal comment (lines 39-41) correctly documents that `AddDelayedMessageScheduler` /
`UseDelayedMessageScheduler` are gone; confirmed no residual reference to the delayed scheduler or the
`rabbitmq_delayed_message_exchange` plugin remains in the file. No defect.
**Fix:** None required — recorded for completeness of the gate-removal verification.
