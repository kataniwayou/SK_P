# Phase 24.1 — Locked Decisions (CONTEXT)

Derived from an extended design review after Phase 24 closed as **FAILED verification**. Records the
decisions, the alternatives considered-and-rejected, and the load-bearing invariants. Downstream
agents MUST read this + 24.1-SPEC.md before implementing.

## Origin

Phase 24 FAILED: clean-build suite 4 failed / 331 passed (all 4 in `Features.Orchestration.*`,
in-scope), + WR-01 (High, plugin-dependent redelivery) + WR-02 (Med, null-terminal NRE). The four
failures traced to a Stop cleanup-ordering regression (3) and an orphan test asserting superseded
Phase-22 behavior (1).

## Locked decisions

### D-24.1-01 — Dedup on L2-root existence (data = the dedup gate)
Start/Stop dedup on whether the L2 ROOT key exists, NOT on parent-index membership. Rationale: when
the gate and the data are the same key, a failed/half write leaves no false claim, so every crash
state self-heals on retry. (Parent-index-as-gate was rejected — see Rejected A.)

### D-24.1-02 — Parent index = derived record, compensated on failure
The parent SET is enumeration-only (boot scheduling). Start: SADD + write L2; on L2-write fault →
SREM (compensate), no publish. Stop: SREM + delete L2; on L2-delete fault → SADD (compensate), no
publish. Compensation is BEST-EFFORT — a fault during compensation degrades to D-24.1-01's
self-healing (dedup reads the L2 root, not the parent). Hard crash between the two writes is
survivable for the same reason.

### D-24.1-03 — Atomic Stop delete: discover-then-delete, one batch
Stop reads the root → BFS-collects all reachable step keys → deletes root + steps (+ parent SREM) in
ONE `CreateBatch().Execute()`. The root MUST be read before deletion (it is the discovery entry
point). This absorbs Phase 24 locked fix (a) and additionally makes a crash leave all-or-nothing.
Root cause of 3 failing tests: the old code deleted the root FIRST, so cleanup read an absent root
and bailed before deleting steps → per-step leak.

### D-24.1-04 — Orphans cannot form (invariant)
L2 holds only the validated, BFS-reachable graph and is NEVER mutated between Start and Stop
(result path is L1-only, D-08). First-win (D-04) makes a re-Start of an existing root a no-op, so
the Phase-22 "shrink-graph-on-re-Start leaves an orphan step key" path is eliminated. Therefore the
Stop reachable-walk deletes the COMPLETE key set; no unreachable orphan exists. The only residual
(crash mid-Stop-delete) is closed by D-24.1-03's atomic batch. ⇒ the orphan test
(`ReStart_Removes_Orphan_Step`) asserts superseded behavior and is rewritten to first-win semantics.
**Load-bearing:** if a future phase adds L2 mutation on the result path, this invariant breaks and
orphans become possible — revisit then.

### D-24.1-05 — Remove the boot gate; result handling is L1-only and lifecycle-agnostic
Remove `IStartupGate` checks + `GateClosedException` from all consumers, remove
`UseScheduledRedelivery` + the delayed message scheduler, drop the `rabbitmq_delayed_message_exchange`
plugin dependency. Processors send results freely at any time. `ResultConsumer`: L1 hit → advance;
**L1 miss → graceful business-ack** (log + return; never throw, never DLQ) — uniformly for unknown /
stopped-drained / not-yet-hydrated ids. **Accepted tradeoff:** a result arriving in the boot window
before its L1 entry is (re)hydrated is gracefully acked (consumed, not advanced) → an active-workflow
DAG MAY stall on a restart-with-backlog. This SUPERSEDES D-06 (never-drop → best-effort-against-L1)
and retires ORCH-GATE-01. Operator chose this over the loss-free alternatives (see Rejected D/E)
explicitly for maximal simplicity and zero plugin/endpoint-ordering machinery.

### D-24.1-06 — Terminal-step guard (WR-02)
`SelectNext` treats null `NextStepIds` as terminal: `?? Enumerable.Empty<Guid>()`. null/empty = "no
successors" per the contract; the lone unguarded site NRE'd on the normal end-of-branch case.

### D-24.1-07 — Single-replica WebApi (TOCTOU deferred)
Dedup concurrency (two Starts racing the L2-root check) is safe under a single active WebApi replica.
Multi-replica atomic claiming is deferred (would need a distributed lock atop D-24.1-01/02),
consistent with ORCH-SCALE-01.

## Preserved (NOT changed)
- **D-07** — Stop keeps the L1 entry for drain (late results drain via L1 until the next Start of
  that id reloads it). Drain remains lifetime-bounded (FUTURE-STOP-EVICTION still deferred).
- **D-08** — result path is L1-only (no L2 read). STRENGTHENED: with the gate gone, L1 is the SOLE
  result arbiter.
- **ORCH-RESULT-01/02, ORCH-ADVANCE-01/02** — contracts, shared competing-consumer endpoint,
  traversal, dispatch — unchanged.

## Alternatives considered and REJECTED (so they are not re-proposed)

- **Rejected A — dedup on parent-index membership (gate-first):** introduces a claim/commit gap — a
  crash after SADD-before-L2-write leaves a claim with no data → workflow permanently un-startable.
  Reordering (SADD last) breaks dedup AND boot-enumeration. Read-after-write between non-atomic
  writes adds latency, not atomicity. An in-process gate/lock fixes concurrency, not the
  crash-between-writes gap. ⇒ root-key dedup (D-24.1-01) is gap-free; keep it.
- **Rejected B — Option C: L2-as-realtime + L1 read-through + TTL-on-Stop:** elegant and dissolves
  WR-01, but supersedes D-07/D-08/ACK-01, adds a Redis read per result, relocates (not removes) the
  drain question, and forces close-gate handling of lingering TTL'd keys. Redesign-scope, not
  gap-closure. Captured as a possible FUTURE milestone.
- **Rejected C — single-document L2 (steps nested in root, one key per workflow):** cleanest for
  "delete-removes-all" and dissolves orphans, but rewrites the Phase-15 L2 key contract that
  Phases 15/16/22/23/24 depend on (and their tests). Milestone-level blast radius. FUTURE idea.
- **Rejected D — Option A: enable rabbitmq_delayed_message_exchange plugin (keep gate):** zero code
  change but adds a non-default community plugin every environment must ship + version-match; silent
  re-loss if ever missing. Operator chose gate removal instead.
- **Rejected E — Option B: delay result-endpoint start until hydrated (durable-queue buffer):**
  loss-free without the plugin, but keeps gate/hydration ordering machinery. Operator chose the
  simpler gate-removal (D-24.1-05) accepting the boot-window best-effort tradeoff.
