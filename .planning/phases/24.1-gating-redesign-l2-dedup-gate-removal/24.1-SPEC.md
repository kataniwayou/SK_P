# Phase 24.1 — Gating Redesign (close the FAILED Phase 24 verification)

## Why this phase exists

Phase 24 closed-out as **FAILED verification**: a clean-build full suite ran **4 failed / 331 passed**
(all 4 in `BaseApi.Tests.Features.Orchestration.*` — in-scope, not flaky), plus two code-review
findings (WR-01 High, WR-02 Medium). Root causes, established by investigation:

1. **Stop cleanup-ordering regression** — `OrchestrationService.StopAsync` deletes the L2 root
   BEFORE `StopCleanupAsync`, which reads the root to discover per-step keys and early-returns when
   the root is absent → per-step keys leak → 3 tests fail (`StopGateFacts.Stop_AllExist_204`,
   `StopGateFacts.Stop_MixedBatch_DeletesPresent_NoOpAbsent_204`,
   `StopScanFacts.Stop_AfterStart_RemovesRootAndStep_KeepsProcessor`).
2. **Orphan-step test asserts superseded Phase-22 behavior** —
   `StartLoopFacts.ReStart_Removes_Orphan_Step` expects a re-Start to overwrite-and-GC a shrunk
   graph; first-win (D-04) makes re-Start a no-op, so the orphan-GC path is gone by design.
3. **WR-01 (High)** — gate-closed redelivery depends on the `rabbitmq_delayed_message_exchange`
   broker plugin, absent from compose → a result arriving during boot dead-letters to `_error`.
4. **WR-02 (Medium)** — `StepAdvancement.SelectNext` lacks a null-guard on `NextStepIds`; a
   terminal step (null `NextStepIds` = "no successors") NREs instead of acking cleanly.

After an extended design review, the resolution is a **targeted gating redesign**, not a pure
patch. This SPEC locks it.

## Goal

Make Start/Stop duplicate-suppression and lifecycle **self-healing and crash-survivable**, make Stop
**actually delete** the full L2 key set atomically, **remove the boot gate** entirely (results are
handled L1-only and gracefully regardless of boot/start/stop state), and fix the terminal-step
advancement guard — so the full suite goes green from a clean build and the orchestrator no longer
depends on a non-default RabbitMQ plugin.

## Requirements (locked)

### R1 — Dedup on L2-root existence (not parent-set membership)
- **Target:** WebApi Start/Stop dedup on whether the **L2 root key exists** (the data IS the dedup
  gate). Start: if root absent → proceed; if present → skip (first-win). Stop: if root present →
  proceed; if absent → no-op.
- **Why:** the dedup gate and the data are the same key → a failed/half write leaves no false claim
  → every crash state is self-healing on retry (Start re-checks the root, Stop re-checks the root).
- **Acceptance:** a 2nd Start for an existing root does not overwrite/republish; a 2nd Stop for an
  absent root is a clean no-op (NOT 422).

### R2 — Parent index is a derived record, compensated on failure
- **Target:** The parent index SET is enumeration-only (boot scheduling). Start: `SADD` parent +
  write L2; **if the L2 write throws, `SREM` parent (compensate) and do NOT publish.** Stop: `SREM`
  parent + delete L2; **if the L2 delete throws, `SADD` parent (compensate) and do NOT publish.**
- **Why:** keeps the parent index consistent with L2 in ordinary (caught) failures; a hard crash
  degrades to R1's self-healing (dedup reads the L2 root, not the parent).
- **Acceptance:** an injected L2-write fault on Start leaves the parent index WITHOUT the id and
  publishes nothing; an injected L2-delete fault on Stop leaves the parent index WITH the id and
  publishes nothing. Compensation is best-effort (a fault during compensation degrades to R1).

### R3 — Atomic Stop delete (discover-then-delete, one batch)
- **Target:** Stop deletes the **complete** L2 key set for a workflow: read the root → BFS-collect
  all reachable per-step keys → delete root + all step keys (+ `SREM` parent) in **one
  `CreateBatch()` `Execute()`**. The root MUST be read before it is deleted (it is the entry point
  to discover the step keys). Absorbs Phase 24 locked fix (a).
- **Why:** the current delete-root-first ordering destroys cleanup's input → per-step keys leak. One
  atomic batch also makes a crash leave either everything or nothing (no half-delete residue).
- **Acceptance:** after Stop, the root AND every per-step key are gone; the processor key is RETAINED
  (cleanup never touches processor keys); a snapshot is net-zero for that workflow.

### R4 — Orphan step keys cannot form (invariant, documented)
- **Target:** Record as a load-bearing invariant: **L2 holds only the validated, reachable graph and
  is never mutated between Start and Stop**, therefore the Stop reachable-walk deletes the COMPLETE
  key set and no unreachable orphan can exist. The Phase-22 graph-shrink-on-re-Start orphan path is
  eliminated by first-win (re-Start of an existing root is a no-op; to reshape, Stop→edit→Start).
- **Why:** the writer projects only the BFS-reachable validated snapshot; the result path is L1-only
  (no L2 mutation); first-win forbids overwrite-in-place. The only residual (crash mid-Stop-delete)
  is closed by R3's atomic batch.
- **Acceptance:** `StartLoopFacts.ReStart_Removes_Orphan_Step` is REWRITTEN to assert first-win
  semantics (a re-Start of an existing workflowId does not overwrite/re-project — it is a deduped
  no-op); a SPEC/CONTEXT note records the immutability invariant.

### R5 — Remove the boot gate; result handling is L1-only and lifecycle-agnostic
- **Target:** REMOVE `IStartupGate.IsReady` checks + `GateClosedException` from all consumers; REMOVE
  `UseScheduledRedelivery` + `AddDelayedMessageScheduler`/`UseDelayedMessageScheduler`; REMOVE the
  `rabbitmq_delayed_message_exchange` plugin dependency. Processors send `ExecutionResult` freely at
  any time. `ResultConsumer`: L1 hit → advance; **L1 miss → graceful business-ack (log + return,
  never throw, never DLQ)** — uniformly for unknown / stopped-drained / not-yet-hydrated ids.
- **Why:** the plugin-backed redelivery is the WR-01 failure; removing the gate kills the plugin
  dependency and the DLQ failure mode. An L1 miss is the DEFINED graceful outcome, not an error.
- **Accepted tradeoff (supersedes D-06):** a result arriving in the boot window before its L1 entry
  is (re)hydrated is gracefully acked (consumed, not advanced) — an active-workflow DAG MAY stall on
  a restart-with-backlog. This relaxes the never-drop guarantee to **best-effort-against-L1**.
- **Acceptance:** no consumer references `IStartupGate`/`GateClosedException`/scheduled-redelivery;
  no `UseDelayedMessageScheduler`; a result for an id absent from L1 is acked (no throw, no `_error`)
  regardless of lifecycle state; an infra fault from `Send` still propagates to bounded retry.

### R6 — Terminal-step advancement guard (WR-02)
- **Target:** `StepAdvancement.SelectNext` treats `null` `NextStepIds` as terminal (no successors),
  consistent with the contract and every sibling traversal site:
  `foreach (var nextId in completed.NextStepIds ?? Enumerable.Empty<Guid>())`.
- **Why:** `null` `NextStepIds` is the contract-defined "terminal step" representation; the lone
  unguarded site NREs on the normal end-of-branch case → would dead-letter instead of acking.
- **Acceptance:** a result for a terminal step (null/empty `NextStepIds`) yields zero dispatches and
  acks cleanly (no NRE, no throw).

### R7 — Clean-build green
- **Target:** A from-scratch (`dotnet clean` then `dotnet test SK_P.sln -c Debug`) full suite passes:
  exit 0, 0 failed, with the real-stack E2E running live.
- **Acceptance:** authoritative clean-build run is GREEN; the 4 previously-failing tests pass
  (3 via R3, 1 rewritten via R4); no compile errors from a clean build.

## Boundaries

**In scope:** R1–R7 — WebApi Start/Stop dedup+compensation+atomic-delete; orchestrator gate removal;
result-consumer L1-only graceful handling; `SelectNext` terminal guard; test reconciliation; clean
removal of gate/redelivery/scheduler wiring and the plugin dependency.

**Out of scope (unchanged from Phase 24):** the processor implementation that emits results
(future Processor milestone — this phase still only consumes); `FUT-REQRESP-01`; multiple-results-
per-step; duplicate-delivery dedup (ORCH-SCALE-01); L1 eviction of stopped workflows
(FUTURE-STOP-EVICTION — drain remains lifetime-bounded via kept-L1, D-07); single-document L2 model
and L2-only/read-through result paths (explored, NOT adopted — captured as future ideas).

**Single-replica WebApi assumption (locked):** dedup concurrency (TOCTOU on the L2-root check) is
safe under a single active WebApi replica (in-process serialization). Multi-replica atomic claiming
is explicitly deferred (would need a distributed lock on top of R1/R2) — consistent with
ORCH-SCALE-01.
