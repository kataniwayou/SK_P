<task number="1" tdd="false" name="WebApi Start/Stop: L2-existence dedup + parent compensation + atomic Stop delete">
**Requirements: R1, R2, R3.**

`src/BaseApi.Service/Features/Orchestration/OrchestrationService.cs`:

**StartAsync (R1/R2):**
- Keep the existing `ExistenceCheckAsync` (Postgres 404 gate) and per-id loop.
- Dedup remains on the L2 ROOT existence probe (`KeyExistsAsync(Root(id))`) — already present; keep
  the first-win `if (alreadyPresent) continue;`. (This is R1 — dedup gate = the data key.)
- **Add parent-index compensation (R2):** the writer already `SADD`s the parent index inside its
  batch. For a per-id Start that PROCEEDS, if the L2 write (`_redisProjectionWriter.UpsertAsync`)
  throws, ensure the parent index does NOT retain this id: wrap so that on a write fault the id is
  `SREM`'d from the parent index (compensate) before rethrow, and it is excluded from `started`
  (so nothing publishes for it). Tag the RedisException op name as today (`"UpsertAsync"`).
  - NOTE: because `UpsertAsync` does root+parent+steps in ONE batch, a batch failure already leaves
    nothing written (atomic) — but the compensation `SREM` is the belt-and-suspenders for any
    partial/older path and makes the intent explicit and testable. Keep it best-effort (a fault in
    the SREM itself degrades to R1 self-healing; do not mask the original fault).

**StopAsync (R1/R3/R2):**
- Dedup on L2 ROOT existence: replace the lead `var rootDeleted = await db.KeyDeleteAsync(Root(id))`
  with a **probe** `var rootPresent = await db.KeyExistsAsync(Root(id))`. Keep `if (!rootPresent)
  continue;` (first-win no-op — NOT 422).
- For a PRESENT root, call `_cleanup.StopCleanupAsync(id, ct)` which (per Task 2) now reads the root,
  BFS-collects step keys, and deletes root + steps + SREMs parent in ONE batch. Add id to `stopped`.
- **Compensation (R2):** if `StopCleanupAsync` throws (L2 delete fault), the parent index may have
  been removed by the cleanup batch — ensure the id is `SADD`'d back (compensate) before rethrow and
  excluded from `stopped` (nothing publishes). Tag op name `"KeyDeleteAsync"`.
- Update the loop's inline comment ("KeyDeleteAsync on the root FIRST" → "KeyExistsAsync probe;
  cleanup owns the atomic delete").

Acceptance greps: no `KeyDeleteAsync(RedisProjectionKeys.Root(` as the lead Stop op; `KeyExistsAsync`
probe present on both Start and Stop; compensation SREM/SADD present on the fault paths.

Commit: `fix(24.1): WebApi L2-existence dedup + parent compensation + probe-not-delete Stop`
</task>

<task number="2" tdd="false" name="RedisL2Cleanup: atomic discover-then-delete (root + steps + SREM in one batch)">
**Requirements: R3, R4.**

`src/BaseApi.Service/Features/Orchestration/Projection/RedisL2Cleanup.cs`:
- The routine already: SREMs parent (hoisted), GETs root, early-returns if absent, BFS-collects step
  keys, then batch-deletes root (unconditional) + step keys. This is correct ONCE the caller stops
  pre-deleting the root (Task 1 probe). Verify the SREM + delete are in the SAME batch as the step
  deletes (one `CreateBatch()` → one `Execute()`); if the SREM is currently a separate awaited op
  ABOVE the early-return, that is acceptable (it must stay idempotent), but the root+steps deletes
  MUST be one atomic batch (R3). Leave the hoisted SREM as-is (idempotent GC) — do NOT double-SREM.
- Add/confirm a code comment recording the R4 invariant: "L2 holds only the validated reachable
  graph and is never mutated between Start and Stop; the reachable walk therefore deletes the
  COMPLETE key set — no unreachable orphan can exist (first-win forbids overwrite-in-place)."
- No behavioral change is expected here beyond confirming atomicity; the FIX is the caller ordering
  (Task 1). If the SREM must move INTO the delete batch to be atomic with it, do so and remove the
  hoisted SREM (no double-SREM).

Commit: `fix(24.1): confirm atomic root+steps delete in RedisL2Cleanup + record reachability invariant`
</task>
