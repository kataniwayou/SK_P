<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| L1 data -> queue:{processorId} dispatch target | A processorId taken from L1 (originally L2-author-supplied) becomes a queue address. |
| Per-workflow SemaphoreSlim stripe | Concurrency primitive that, if leaked/never-released, exhausts resources (Pitfall 5). |
| Quartz self-reschedule | A reschedule overlapping trigger completion could double-fire (Pitfall 4). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-23-05 | Elevation/Spoofing | queue:{processorId} dispatch address | mitigate | processorId is read from L1 which is hydrated read-only from L2 (WebApi-owned, already-trusted at write time); the orchestrator never derives the address from inbound bus/user data. `:D` GUID formatting is consistent with L2ProjectionKeys, so a corrupt processorId yields at worst a dead-letter to a non-existent queue, not cross-tenant injection beyond the existing trust model (ORCH-SCALE/ACK). No new trust boundary is crossed. |
| T-23-06 | Denial of Service (resource exhaustion) | per-workflowId SemaphoreSlim stripe | mitigate | Stripe uses `Wait(0)` (drop-if-held, never blocks) so no thread parks; `Release` is paired in a `finally` by callers (Plan 04); semaphore count is bounded by the finite parent-index workflow population and never-disposed handles from a Wait(0)-only semaphore hold no kernel resource (Pitfall 5). Acceptance asserts no bare blocking `Wait()`. |
| T-23-07 | Tampering (double-dispatch) | Quartz self-rescheduling one-shot | mitigate | `[DisallowConcurrentExecution]` prevents two Execute()s for one jobKey; reschedule adds a trigger to the existing job (not a re-add); `DeleteJob` removes job+triggers atomically on teardown (Pitfall 4a/b/c). |
</threat_model>
