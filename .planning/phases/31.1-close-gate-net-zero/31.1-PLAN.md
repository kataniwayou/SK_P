# Phase 31.1 — Close-Gate Redis Net-Zero (NET-ZERO-31)

## Why

Phase 31's `phase-31-close.ps1` redis `--scan` triple-SHA fails (BEFORE≠AFTER). Root cause
(diagnosed in 31-VERIFICATION.md): RealStack E2E tests create cron `* * * * *` workflows that
`WorkflowFireJob` **self-reschedules**, so the live `sk-orchestrator` fires them every minute
forever. Phase 31's `skp:flag:{H}` names embed the per-fire `correlationId`, so every fire mints
new key names → the `--scan` key-name set churns unboundedly. No E2E test stops its workflow.
(Prior phases were immune: content-addressed data-key names are deterministic.)

The permanent-flag-key production bug (`SET XX` clearing TTL) was already fixed in Phase 31
(`2a4837f`, `keepTtl`); flag/data keys are now bounded to the 300 s `ExecutionDataTtl`/`FlagTtl`.
This phase stops the churn so the bounded keys actually drain to net-zero.

## Goal

`phase-31-close.ps1` passes clean: 3×GREEN + ALL THREE triple-SHA invariants (psql, redis, rabbitmq)
BEFORE=AFTER, including the new `skp:data:*`/`skp:flag:*` namespaces.

## Tasks

### Task 1 — Stop-workflow-in-teardown across the 5 RealStack E2E tests
Each RealStack E2E test's nested factory gains a `WorkflowIdsToStop` list (mirroring the existing
`L2KeysToCleanup`/`ParentIndexMembersToSrem` discipline), populated right after each `Start`, and
drained in `DisposeAsync` via `POST /api/v1/orchestration/stop` (the existing seam →
`StopOrchestration` → `UnscheduleOnlyAsync` removes the Quartz job). Runs even when an assertion
throws (`await using`), so no workflow is left firing.

Files:
- `tests/BaseApi.Tests/Orchestrator/SampleRoundTripE2ETests.cs`
- `tests/BaseApi.Tests/Orchestrator/CorrelationPropagationE2ETests.cs`
- `tests/BaseApi.Tests/Orchestrator/MetricsRoundTripE2ETests.cs`
- `tests/BaseApi.Tests/Orchestrator/IdempotentExactlyOnceE2ETests.cs`
- `tests/BaseApi.Tests/Observability/OrchestrationLogsE2ETests.cs`

### Task 2 — Close-gate settle-drain before the AFTER snapshot
`scripts/phase-31-close.ps1`: after run 3, before capturing the AFTER redis snapshot, poll until
`skp:data:*` + `skp:flag:*` counts return to their BEFORE baseline (timeout = TTL + slack, ~330 s).
With every workflow stopped (Task 1), the bounded keys drain within the 300 s TTL. A key that never
drains (e.g. a future permanent-key regression, or a workflow left firing) still fails the gate —
leak detection is preserved, not weakened.

### Task 3 — One-time purge of already-leaked jobs + live gate to GREEN
Quartz is in-memory (RAMJobStore), so restart `sk-orchestrator` to clear the leaked in-memory jobs
from prior runs (hydration reschedules only workflows still present in L1), clear residual
`skp:flag:*`/`skp:data:*`, then run `phase-31-close.ps1` to a clean pass.
