<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| processor → L2 (index key) | The forward cleanup tail deletes the index `L2[messageId]`; the keeper (INJECT) also touches L2 for the same escalated item — a race window on the index key. |
| processor → keeper / redelivery → Recovery | When cleanup is skipped, the keeper completes the escalated item and a later Recovery pass / TTL reclaims the keys. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-69-02 | Tampering / DoS (correctness — processor/keeper race on the index key) | Forward cleanup tail `DeleteTerminalAsync` (`ProcessorPipeline.cs:309`) | mitigate | Unconditional cleanup deletes the index while the keeper is still completing an escalated item → race on the index key (the keeper's work could be reclaimed mid-flight, or a redelivery re-enters Forward and reprocesses). Mitigated by gating the tail on `!escalated`: when any item escalated, the index + input keys are LEFT intact for the keeper + a later Recovery pass / index-TTL. Proven by `EscalatedItem_SkipsCleanup` (`db.DidNotReceive().KeyDeleteAsync(RedisKey[])`). |
| T-69-05 | DoS (key leak if skipped cleanup never reclaimed) | Skipped-cleanup keys (index + input) | accept | If no redelivery ever arrives, the keys are reclaimed by the index's random TTL (`SlotTtl()` ∈ [ttl, 2×ttl], born inside the Plan-01 atomic write) and the data TTL — bounded, never an unbounded leak. The Recovery pass (existing `PipelineRecoveryFacts`) idempotently reclaims on redelivery (GATE-02). Backstopped by TTL; no further mitigation. |
</threat_model>
