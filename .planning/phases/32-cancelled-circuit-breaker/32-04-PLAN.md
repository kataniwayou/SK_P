<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| broker тЖТ processor consumer | Untrusted dispatch message crosses here; the consumer reads/writes the broker-internal Redis L2. |
| processor тЖТ Redis L2 marker | The `skp:cancelled:*` marker SET halts the whole workflow тАФ tampering (forging/clearing) is a halt/un-halt surface. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-01 | Elevation of Privilege | breaker catch (`GetRetryAttempt()==Limit` gate) | mitigate | Catch fires ONLY at the pinned exhaustion boundary; business `ProcessAsync` throws are caught earlier (:123/:129) and never trip тАФ `BreakerTriggerFacts` asserts a business throw sets no marker and re-throws nothing (prevents a business failure escalating into a whole-workflow halt). |
| T-32-02 | Tampering | `skp:cancelled:{workflowId:D}` marker SET | accept | The marker is written/read on the broker-internal Redis, same trust boundary as the existing `skp:flag:*`/`skp:data:*` dedup keys; an attacker with Redis write access could forge/clear it, but that is the pre-existing infra trust boundary (network/broker-isolated Redis), not a new surface. Documented-accept per security_threat_model. |
| T-32-04 | Information Disclosure | WARN trip log | mitigate | The four ids (workflowId/stepId/processorId/H) are server-minted Guids/hash passed as structured MESSAGE-TEMPLATE args, never interpolated into the template string (T-18-04) тАФ no log-injection / tainted-template surface. |
| T-32-R4 | Denial of Service | marker-write-fault edge | accept | If the marker `SET` itself faults (Redis down) the in-flight marker isn't set; this lands in the same bus/redis-down `_error` backstop the design already accepts (Risk R4). No mitigation required. |
</threat_model>
