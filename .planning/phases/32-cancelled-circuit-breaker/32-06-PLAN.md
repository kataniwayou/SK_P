using MassTransit;
using Messaging.Contracts;

namespace Orchestrator.Consumers;

/// <summary>
/// Phase 32 (req-4 / D-06): the future-fire stop. Consumes the MassTransit-auto-published
/// <c>Fault&lt;EntryStepDispatch&gt;</c> (from the processor breaker's re-throw exhausting UseMessageRetry)
/// on a per-replica InstanceId+Temporary fan-out endpoint (mirrors Start/Stop, NOT shared orchestrator-result).
/// Extracts WorkflowId from Fault.Message, reuses the idempotent keep-L1 <see cref="WorkflowLifecycle.UnscheduleOnlyAsync"/>.
/// Only the schedule-owning replica acts; others no-op (absent-from-L1 business no-op inside lifecycle).
/// D-13: reads/writes NO flag[H] key and NO L2 marker (the processor already set the marker; the halt is
/// naturally idempotent — DO NOT seed flag[resultH]=Pending here, unlike the Completed-result pre-write).
/// </summary>
public sealed class FaultUnscheduleConsumer(
    WorkflowLifecycle lifecycle,
    ILogger<FaultUnscheduleConsumer> logger) : IConsumer<Fault<EntryStepDispatch>>
{
    public async Task Consume(ConsumeContext<Fault<EntryStepDispatch>> context)
    {
        var workflowId = context.Message.Message.WorkflowId;   // Fault.Message IS the original EntryStepDispatch
        logger.LogWarning("Fault halt — unscheduling workflow {WorkflowId}", workflowId);
        // Reuse the idempotent keep-L1 unschedule (D-06). Absent-from-L1 ⇒ business no-op inside lifecycle.
        // NO flag[H] read/write (D-13). NO L2 marker write (the processor already set it).
        await lifecycle.UnscheduleOnlyAsync(workflowId, context.CancellationToken);
    }
}

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| broker (published Fault) → per-replica fan-out endpoint | A published `Fault<EntryStepDispatch>` fans out to every replica's temp queue; a forged Fault could unschedule arbitrary jobs (fault-flood DoS). |
| fault consumer → Quartz/L1 | The unschedule deletes a Quartz job; only the schedule-owning replica's L1 has the JobId. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-04 | Spoofing/DoS | forged `Fault<EntryStepDispatch>` publish (fault-flood) | accept | Publishing onto the broker requires broker credentials — the same network/broker trust boundary that already protects every Send/Publish in the system; an attacker with publish access can already disrupt the bus. The unschedule is idempotent + keyed on a server-minted WorkflowId resolved from L1 (absent ⇒ no-op), so a forged fault for an unknown workflow is a harmless no-op. No new app-level auth surface; mitigated by the existing broker trust boundary (stated explicitly per security_threat_model). |
| T-32-04b | Tampering | duplicate/replayed fault delivery | mitigate | The halt is naturally idempotent — `UnscheduleOnlyAsync` is a jobId-addressed DeleteJob with an absent-from-L1 no-op; `FaultIdempotencyFacts` asserts two deliveries == one end state. No CAS gate needed (D-13). |
| T-32-08b | Elevation of Privilege | fault consumer on wrong (shared) endpoint | mitigate | `EndpointName = "orchestrator"` + `.Endpoint(InstanceId/Temporary)` puts it on the per-replica fan-out queue so EVERY replica (incl. the schedule owner) receives the fault — NOT the shared `orchestrator-result` competing-consumer queue where only one replica would get it and could miss the owner (Pitfall 5). Acceptance asserts the registration shape. |
| T-32-08c | Tampering | fault consumer seeding flag[H]=Pending | mitigate | The consumer takes no Redis dependency (lifecycle + logger only) — it CANNOT read/write flag[H] or the marker by construction (D-13 / Pitfall 4); `FaultIdempotencyFacts` asserts the ctor shape. |
