## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| startup thread → Redis (L2) | the orchestrator's inline write crosses into the shared keyspace via the shared writer |
| pre-identity window | no processorId exists → no key (accepted gap, D-02) |
| Redis fault → resolution | a Redis failure must not abort identity/definition resolution |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-60-12 | Spoofing / poisoning | per-instance liveness key during startup | mitigate | Key via `L2ProjectionKeys.PerInstance(procId, instanceId)` only; `procId` guarded non-null (D-02); status always built through `ProcessorLivenessEntry.Create` (cannot fabricate a Healthy status while schemas unresolved). |
| T-60-13 | Denial of service | Redis fault crashing the host or aborting resolution | mitigate | Write routed through the shared writer's `try/catch → log-and-continue`; resolution proceeds (verified by the resilience fact: still reaches Healthy on a dead Redis). |
| T-60-14 | Denial of service / false-absent | slow-backoff startup write gap (up to BackoffCap 30s) outliving the TTL | mitigate | Startup entry records interval 30 → TTL = max(30*2, 30) = 60, comfortably outliving the worst-case 30s backoff gap → STATE-03 "never absent" preserved. |
| T-60-15 | Repudiation / desync | old `ProcessorProjection` flat key written during startup | mitigate | Orchestrator never wrote the old key; fact asserts `KeyExistsAsync(L2ProjectionKeys.Processor(id))` is false (D-05). |
| T-60-16 | Information disclosure | the `unhealthy` summary's `configSchema` diagnostic field (Open Q1) | accept | Diagnostic-only; status is `Unhealthy` throughout startup regardless of the config field's value; no secret content. |
</threat_model>
