## OWNERSHIP DECISION: Model A1 (RESOLVED — rationale below)
**A1 is chosen.** The holder owns ONLY provider #2; the HOST builds provider #1 (the placeholder) via the unchanged shared `AddBaseConsoleObservability` path; at swap time the holder disposes the host's provider #1 and installs its own #2. The DI container disposes provider #1 again at host shutdown — a SAFE no-op (`MeterProvider.Dispose()` is idempotent by OTel .NET 1.15.3 contract).

Rationale:
1. **D-06 (locked) demands the shared `AddBaseConsoleObservability` path be used LITERALLY UNCHANGED**, and CONTEXT.md frames the processor-specific behavior as "ONLY the later swap." A1 touches zero shared-path lines; A2 would add a host-provider-removal line in `AddBaseProcessor` and have the holder build BOTH providers.
2. **Provider #1 genuinely IS the host's shared provider:** `BaseProcessorServiceCollectionExtensions.cs:122` already does `services.ConfigureOpenTelemetryMeterProvider(mp => mp.AddMeter(ProcessorMetrics.MeterName))` — the `BaseProcessor` meter is attached additively to the host MeterProvider that the shared path built (placeholder resource + MassTransit + runtime + bare AddOtlpExporter). The holder reuses this as #1.
3. **The only A1 risk — double-dispose of #1 at shutdown — is provably safe** (RESEARCH Assumption A2: OTel .NET disposes are idempotent; the second Dispose is a no-op). The hermetic test (Task 3) explicitly proves dispose-of-#1 happens on swap and a second Dispose does not throw.

**Model A1 shape:** the host builds provider #1; the holder receives it via ctor as the initial `_current` and only ever BUILDS provider #2 (in `SwapTo`). Fields + ctor:
```csharp
using OpenTelemetry;
using OpenTelemetry.Metrics;
using OpenTelemetry.Resources;
using MassTransit.Monitoring;

namespace BaseProcessor.Core.Observability;

/// <summary>
/// MLBL-03 (D-02/D-03, Model A1): bridges the frozen-at-Build metrics Resource to the async DB
/// identity. The HOST builds provider #1 with the appsettings placeholder resource (shared path,
/// unchanged — D-06); on identity-resolve in Loop A this holder builds provider #2 with the
/// DB-sourced service.name and disposes provider #1. Meters are NOT recreated — provider #2
/// re-subscribes by meter NAME (BaseProcessor / MassTransit / runtime). The single resolved
/// service.instance.id (Phase 30 D-10) is captured ONCE (ctor param) and reused across the swap so
/// logs + metrics #1 + metrics #2 carry the identical value. Dispose of provider #1 at swap +
/// the DI container's shutdown Dispose are both safe (MeterProvider.Dispose is idempotent).
/// </summary>
public sealed class MeterProviderHolder : IDisposable
{
    private readonly string _instanceId;     // captured ONCE — never re-resolve (Pitfall 4 / Phase 30 D-10)
    private readonly string _serviceVersion; // appsettings Service:Version — kept as service.version attr (D-07)
    private MeterProvider _current;          // starts as the host-built provider #1 (A1)

    public MeterProviderHolder(MeterProvider hostProvider, string instanceId, string serviceVersion)
    {
        _current        = hostProvider;      // A1: provider #1 is the host's; holder will swap to #2
        _instanceId     = instanceId;
        _serviceVersion = serviceVersion;
    }

    private MeterProvider Build(string serviceName) =>
        Sdk.CreateMeterProviderBuilder()
           .ConfigureResource(r => r
               .AddService(serviceName: serviceName, serviceVersion: _serviceVersion)   // service.version kept (D-07)
               .AddAttributes(new[] {
                   new KeyValuePair<string, object>("service.instance.id", _instanceId) // SAME id across swap (Pitfall 4)
               }))
           .AddMeter(ProcessorMetrics.MeterName)               // "BaseProcessor" — const, NOT a literal
           .AddMeter(InstrumentationOptions.MeterName)         // "MassTransit"
           .AddRuntimeInstrumentation()
           .AddOtlpExporter())                                 // BARE — inherits OTEL_EXPORTER_OTLP_ENDPOINT (do NOT hardcode)
           .Build();

    /// <summary>Called from Loop A immediately after SetIdentity, BEFORE the queue-bind / MarkHealthy.</summary>
    public void SwapTo(string resolvedServiceName)
    {
        var next  = Build(resolvedServiceName);   // (1) build #2 first — a live provider exists before #1 dies
        var prior = _current;
        _current  = next;                          // (2) repoint BEFORE disposing prior
        prior.ForceFlush(milliseconds: 5000);      // (3) push the placeholder window's in-flight batch
        prior.Dispose();                           // (4) flush+shutdown reader, release OTLP gRPC channel
    }

    public void Dispose() => _current?.Dispose();  // host shutdown — idempotent if also disposed by DI (A1 safe)
}
```
Notes for the executor:
- `Sdk.CreateMeterProviderBuilder()` yields a provider NOT owned by the host (standalone) — that is exactly why provider #2 is safe to own here.
- The swap ordering (build #2 → repoint → ForceFlush → Dispose #1) is load-bearing for race-safety (Pitfall 2): `_current` is ALWAYS a live provider; runtime/MassTransit meters always write to the current.
- `AddRuntimeInstrumentation()` requires `using OpenTelemetry.Metrics;` (extension method); `InstrumentationOptions` is `MassTransit.Monitoring.InstrumentationOptions`.
- Do NOT re-implement the `POD_NAME ?? HOSTNAME ?? MachineName ?? GUID` precedence here — that would be a 4th IN-03 drift site. Capture the value (ctor param).
