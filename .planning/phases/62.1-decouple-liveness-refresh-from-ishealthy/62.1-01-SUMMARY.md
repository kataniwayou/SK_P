# Phase 62.1 Plan 01: Decouple Liveness Refresh from IsHealthy Summary

Replaced the Gate-A-clash terminal `return;` in `ProcessorStartupOrchestrator` with a cancellation-safe
unhealthy-refresh loop that re-stamps the same static unhealthy per-instance L2 entry every `IntervalSeconds`
(10s) until shutdown — fixing G-62-01, where a clashed-but-alive replica's L2 key TTL-expired (→ absent) and
its L1 record went stale (→ the self-watchdog falsely reported "liveness loop stale").

## What Was Built

**Task 1 — interval-parameterized `WriteUnhealthyAsync`** (commit `b1ede90`)
- Added an optional `int? recordedIntervalSeconds = null` parameter to the private `WriteUnhealthyAsync`.
- The `interval:` argument of `ProcessorLivenessEntry.Create(...)` now reads
  `recordedIntervalSeconds ?? options.Value.StartupIntervalSeconds` — existing callers (Loop A line ~128,
  Loop B line ~173, the clash's first stamp) are byte-unchanged and still record `30`; a new caller can pass
  `IntervalSeconds` (10).
- Updated the D-02 XML-doc guard note to record the interval is now parameterizable (default 30; refresh 10).

**Task 2 — Gate-A-clash refresh loop** (commit `35e6d71`)
- Kept the `logger.LogError(...)`, the first `WriteUnhealthyAsync(configOutcomeOverride: SchemaOutcome.Fail)`
  initial stamp, and `gate.MarkReady()` exactly as before (each fires once; `MarkHealthy` + the bind are still
  NOT reached on the clash path).
- DELETED the bare `return;` and replaced it with:
  `var refreshPeriod = TimeSpan.FromSeconds(options.Value.IntervalSeconds);` then a
  `while (!stoppingToken.IsCancellationRequested)` loop that `await Task.Delay(refreshPeriod, clock, stoppingToken)`
  (catch `OperationCanceledException` → `return`, D-06) then calls
  `WriteUnhealthyAsync(configOutcomeOverride: SchemaOutcome.Fail, recordedIntervalSeconds: options.Value.IntervalSeconds)`.
- A trailing `return;` covers the already-cancelled-before-first-delay edge — clean exit, no fall-through.

Net effect: the clashed replica stays `present + status=Unhealthy` in L2 (STATE-03 — the orchestration-start
gate still BLOCKS it on `status`, D-05 unchanged) and its L1 `Current.Timestamp` advances every interval, so
`LivenessWatchdogHealthCheck` reads it as live (PROBE-02), not falsely "loop stale". The recorded `interval=10`
drives TTL `max(10*2, 30)=30`, so the key never expires while the process is alive.
