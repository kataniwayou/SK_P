# Phase 62.1 Plan 02: Decouple Liveness Refresh from IsHealthy ‚Äî Hermetic Test Coverage Summary

Added `ClashRefreshFacts` ‚Äî a hermetic fact class that locks the Plan-01 G-62-01 fix against regression and
proves the decoupling (timestamp-fresh, status-Unhealthy) WITHOUT a live container lifecycle. A config-schema
definition declaring `Mode` as a string-enum CLASHES the processor's `GateAStubConfig` CLR enum (rule-table
row #13), driving `ProcessorStartupOrchestrator` down the terminal Gate-A-clash path ‚Äî where the Plan-01
refresh loop now re-stamps the SAME static unhealthy per-instance L2 entry every `IntervalSeconds` instead of
the old terminal `return;` that let the key TTL-expire (‚Üí absent) and the L1 record go stale (‚Üí false
"liveness loop stale"). The REAL `ProcessorLivenessWriter` is wired against the `RedisFixture` so the re-SET is
observable in Redis; the watchdog is asserted, never modified.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Over-asserted interval on the INITIAL clash stamp**
- **Found during:** Task 1 (first filtered run)
- **Issue:** The plan's Fact A says "every observed entry has `interval == 10`". The orchestrator's INITIAL
  clash stamp (line ~228, `WriteUnhealthyAsync(configOutcomeOverride: Fail)` with no `recordedIntervalSeconds`
  override) records the startup anchor `interval == 30`; only the REFRESH-loop writes record `interval == 10`.
  Asserting `interval == 10` on the `first` (initial) entry failed (`Expected 10, Actual 30`).
- **Fix:** Split the assertion ‚Äî `AssertClashStatus` (Unhealthy + config=Fail, applied to the initial stamp)
  vs `AssertRefreshEntry` (adds `interval == 10`, applied to every re-SET observed after advancing the clock).
  This is also a STRONGER decoupling proof: the initial stamp records interval=30, the refresh signature is
  interval=10.
- **Files modified:** `tests/BaseApi.Tests/Processor/ClashRefreshFacts.cs`
- **Commit:** `1458e80`

### Process notes (not deviations)

- The `dotnet test ... --filter "FullyQualifiedName~ClashRefreshFacts"` invocation in the plan's `<verify>`
  blocks uses the VSTest filter property, which **Microsoft.Testing.Platform ignores** (`warning MTP0001`) ‚Äî
  it runs the WHOLE suite (~15 min). The equivalent green-proof was obtained with the xUnit-v3 / MTP native
  flag `BaseApi.Tests.exe --filter-class "*ClashRefreshFacts"` (3/3 green in ~5s). No code impact.
