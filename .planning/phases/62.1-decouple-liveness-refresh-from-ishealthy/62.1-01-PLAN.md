<objective>
Fix the Phase-60 product gap G-62-01: the Gate-A-clash terminal path in
`ProcessorStartupOrchestrator` writes ONE `unhealthy` per-instance entry, fires `gate.MarkReady()`,
then `return;`. The per-instance L2 key (TTL 60s on the startup anchor) then expires → the replica
goes ABSENT, and its in-memory L1 record goes stale → `LivenessWatchdogHealthCheck` falsely reports
"liveness loop stale" for a loop that is alive.

Implement CONTEXT D-01 (option **b**): replace that terminal `return;` (after `gate.MarkReady()`)
with a cancellation-safe refresh loop that re-calls
`WriteUnhealthyAsync(configOutcomeOverride: SchemaOutcome.Fail)` every `IntervalSeconds` (10s, D-02)
until shutdown — re-stamping the same static unhealthy entry (config=Fail, other schemas resolved)
with a fresh timestamp and reset TTL.

Purpose: make the liveness-timestamp refresh unconditional on health, conditional only on
"identity resolved + process alive." Status (Unhealthy) stays in the entry; the timestamp keeps
proving the loop is alive — so the gate still BLOCKS the replica (per D-05) but it is now observably
Unhealthy (STATE-03), not absent, and the self-watchdog no longer falsely restarts it (PROBE-02).

Output: a modified `ProcessorStartupOrchestrator.cs` — a ~10-line loop replacing one `return;`, plus
a recorded-interval parameter on `WriteUnhealthyAsync` so the refresh writes record `interval=10`.
No new types. Heartbeat, watchdog, and validator are untouched.
</objective>
