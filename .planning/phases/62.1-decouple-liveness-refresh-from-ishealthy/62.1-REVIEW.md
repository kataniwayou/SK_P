# Phase 62.1: Code Review Report

**Reviewed:** 2026-06-14T00:00:00Z
**Depth:** standard
**Files Reviewed:** 2
**Status:** issues_found (info-only — no bugs or blockers)

## Summary

The phase 62.1 change replaces the terminal `return;` after a Gate-A clash with a cancellation-safe
refresh loop that re-stamps an unhealthy liveness entry every `IntervalSeconds`, plus a new optional
`recordedIntervalSeconds` parameter on `WriteUnhealthyAsync`. I reviewed both the production change in
`ProcessorStartupOrchestrator.cs` and the hermetic `ClashRefreshFacts.cs` test, cross-referencing the
shared `ProcessorLivenessWriter` (TTL derivation, log-and-continue resilience) and
`ProcessorLivenessOptions`.

**Assessment: the change is correct.** The cancellation/shutdown handling is sound, the loop holds no
resources to leak, exception handling is well-scoped, and the shared-writer interaction is unchanged
(blind last-write-wins SET + idempotent SADD + L1 update + swallow-and-warn). The new parameter is a pure
additive default that does not alter any existing caller.

Specifically verified:
- **Cancellation path:** `Task.Delay(refreshPeriod, clock, stoppingToken)` throws `OperationCanceledException`
  on shutdown; the local `catch` returns. The trailing `return` after the loop covers the
  already-cancelled-before-first-delay case. Both exits are clean — no leaked exception (Fact E asserts
  `IsFaulted == false`).
- **No resource leak:** the loop allocates no timers, `CancellationTokenSource`, or disposables. The only
  per-iteration allocation is the `ProcessorLivenessEntry`.
- **Writer interaction:** `WriteUnhealthyAsync` reads `context` state on the orchestrator thread (WR-03
  safe), and `ProcessorLivenessWriter.WriteAsync` swallows all Redis faults, so a refresh write can never
  fault the `BackgroundService`. TTL math `max(10×2, 30) = 30 > 10s` cadence ⇒ the key never expires
  between refreshes.
- **Gate-A still terminal:** only the timestamp/TTL refresh; the `Unhealthy`/`config=Fail` status is static
  and the gate is never re-evaluated (D-11 preserved).

The findings below are all **Info** — defensive-hardening and test-robustness suggestions, none blocking.

## Info

### IN-01: Refresh-loop `catch (OperationCanceledException)` is broader than the single throw site

**File:** `src/BaseProcessor.Core/Startup/ProcessorStartupOrchestrator.cs:244-256`
**Issue:** The `try { await Task.Delay(...) } catch (OperationCanceledException) { return; }` block wraps
only the delay, which is correct today because `WriteUnhealthyAsync` is *outside* the try and the shared
writer never throws. However, the catch swallows *any* OCE — if a future refactor moved the write inside
the try (or `Task.Delay` were swapped for a token-threaded write), a cancellation raised mid-write would
silently exit without the final stamp, and a non-cancellation-triggered OCE would be masked. This matches
the existing `BackoffAsync` pattern, so it is consistent — flagging only as a forward-looking note.
**Fix:** Optionally narrow the intent by filtering on the stopping token, mirroring the established guard
style elsewhere in the file:
```csharp
catch (OperationCanceledException) when (stoppingToken.IsCancellationRequested)
{
    return;
}
```
This makes the "only shutdown cancellation returns" contract (class docstring, line 65) explicit and
keeps an unexpected OCE from being silently absorbed.

### IN-02: Test pumps rely on a fixed 20ms real-time sleep to observe the loop continuation

**File:** `tests/BaseApi.Tests/Processor/ClashRefreshFacts.cs:225,274,314`
**Issue:** After `run.Clock.Advance(...)`, each test does `await Task.Delay(20, ct)` to let the loop's
`Task.Delay` continuation run and re-SET the key before reading. A 20ms wall-clock window is usually
enough on a developer machine, but under CI load or a slow Redis round-trip the continuation +
`writer.WriteAsync` (a real network SET + SADD against `RedisFixture`) may not have completed, making the
`ReadEntryAsync` / TTL / L1 assertions intermittently flaky. This is a test-reliability concern, not a
correctness defect in the code under test.
**Fix:** Prefer polling on an observable post-condition rather than a fixed sleep — e.g. spin until the
entry's `Timestamp` advances past the previously captured value (with a bounded timeout), reusing the
`AdvanceUntilAsync`-style pump. That removes the dependence on a hard-coded delay being "long enough."

### IN-03: Fact E does not assert the index SET key is cleaned up / the loop produced no orphan member

**File:** `tests/BaseApi.Tests/Processor/ClashRefreshFacts.cs:300-330`
**Issue:** The class docstring and `_redis.Track(index)` establish a net-zero teardown contract for both
the per-instance key and the per-test index SET. Facts A/B/C exercise the per-instance key thoroughly, but
no test explicitly asserts the `InstanceIndex` SET contains exactly the expected single member after the
refresh loop has run several iterations (the writer's `SetAddAsync` is idempotent, so this is expected to
hold — but it is unverified). Given the loop now writes the index repeatedly (vs. the old one-shot path),
a one-line assertion would lock in the idempotency guarantee against regressions in the shared writer.
**Fix:** In Facts A/B/C, after the pump loop, add:
```csharp
Assert.Equal(1, await db.SetLengthAsync(index)); // idempotent SADD — exactly one member despite N refreshes
```
