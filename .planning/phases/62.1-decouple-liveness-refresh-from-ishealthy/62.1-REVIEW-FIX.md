### IN-02: Test pumps rely on a fixed 20ms real-time sleep to observe the loop continuation

**Files modified:** `tests/BaseApi.Tests/Processor/ClashRefreshFacts.cs`
**Commit:** 30efcd9
**Applied fix:** Replaced the three fixed `await Task.Delay(20, ct)` pump-waits (Facts A, D, E) with two
polling helpers that spin on an observable post-condition until a bounded wall-clock deadline:
- `PollUntilTimestampAdvancesAsync` — polls the per-instance Redis key until its deserialized `Timestamp`
  advances strictly past the previously captured value (Fact A's re-SET loop).
- `PollUntilL1AdvancesAsync` — polls `ProcessorLivenessState.Current.Timestamp` advancing past the
  pre-advance value (Facts D and E).

Each helper re-advances the `FakeTimeProvider` by one interval per poll iteration, mirroring the existing
`IdentityResolutionFacts.AdvanceUntilAsync` idiom, so a `Task.Delay` parked after the first advance still
fires. No assertion was weakened. Verified: all 3 `ClashRefreshFacts` tests pass green
(`Passed! ... failed: 0, succeeded: 3`).
