# Phase 62.1: Decouple liveness refresh from IsHealthy - Context

**Gathered:** 2026-06-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the Phase-60 product gap (G-62-01) where the processor liveness-timestamp refresh is coupled to
`IsHealthy`, so an **alive-but-unhealthy** replica stops being refreshed after startup. Concretely: a
**Gate-A-clash** replica (config schema does not cover `TConfig`) writes one `unhealthy` entry then the
orchestrator returns (terminal) — the per-instance L2 key TTL-expires (→ **absent**) and the in-memory L1
record goes stale (→ the self-watchdog falsely reports **"liveness loop stale"** for a loop that is alive).

This phase makes the liveness-timestamp refresh **unconditional on health, conditional only on
"identity resolved + process alive."** The two signals are decoupled:
- **Timestamp (refreshed every interval)** = "the loop is alive" → answers *"should K8s restart me?"*
- **Status field (Healthy/Unhealthy)** = "is this replica serving correctly" → answers *"should the
  orchestration-start gate admit me?"*

**In scope:**
- Make the Gate-A-clash terminal path keep refreshing its `unhealthy` per-instance entry every interval
  (the one alive-but-permanently-unhealthy terminal state in the orchestrator).
- Hermetic test coverage proving the refresh keeps the L2 key + L1 record fresh, and that the watchdog
  verdict math is unchanged.

**Explicitly NOT in this phase:**
- **Changing the orchestration-start gate's admit rule** — it correctly already BLOCKS unhealthy replicas
  (admit only Healthy+fresh). The goal is observability (unhealthy-not-absent), not changing what is admitted.
- **Mid-life health re-validation** (`healthy → unhealthy` within a process) — frozen-healthy stays
  (Phase 60 D-14, milestone out-of-scope).
- **The live TEST-01b/TEST-01c re-proof** — handed back to a Phase-62 close-gate re-run (see D-04).
- **Pre-identity coverage** — before Loop A resolves identity there is no `processorId`, hence no key to
  refresh; this exception is inherent and already accepted (Phase 60 D-02). This phase does not change it.

</domain>

<decisions>
## Implementation Decisions

### Fix topology — who refreshes the alive-but-unhealthy timestamp (THE decision)
- **D-01:** **Option b — orchestrator unhealthy-refresh loop.** Replace the terminal `return;` after the
  Gate-A clash (`ProcessorStartupOrchestrator.cs:230`, after `gate.MarkReady()`) with a refresh loop that
  re-writes the **same static unhealthy entry** (`configOutcomeOverride: SchemaOutcome.Fail`, all other
  schemas already resolved) with a **fresh timestamp** every interval, until shutdown. The heartbeat stays
  `IsHealthy`-gated and **unchanged**.
  - **Why b over a:** the entry's `summary` is built from `IProcessorContext`'s schema-resolution state,
    which Phase 60 D-01/WR-03 locked as readable **only on the orchestrator's own single thread** (not
    barrier-safe for the heartbeat thread, and on the clash path `MarkHealthy` is never called so no barrier
    is published). The orchestrator refresh loop reads that state on the same thread that resolved it — safe.
    On the clash path the summary is **static** (config=Fail, rest resolved), so the loop is literally
    "re-stamp the same entry with a new timestamp."
  - **Why NOT a:** option a (identity-gate the heartbeat + have it refresh from the L1 holder) would make the
    heartbeat start writing as soon as identity resolves, **overlapping the orchestrator's own Loop-B
    unhealthy writes** during the startup window — the two-writer situation Phase 60 D-01 deliberately avoided
    with its clean `MarkHealthy` handoff.

### Refresh cadence / recorded interval (drives TTL via Phase-60 D-13)
- **D-02:** The unhealthy-refresh loop uses the **heartbeat cadence** `IntervalSeconds` (**10s**), so each
  refreshed entry records `interval=10` → derived TTL = `max(10×2, Ttl-floor 30) = 30s` (Phase-60 D-13).
  Rationale: a terminally-clashed replica is in **steady-state** (no longer backing off), so the steady-state
  cadence applies — tighter refresh means fresher data for the gate + watchdog and a smaller absent-window
  risk if one write is dropped. (Contrast: the preceding Loop-B unhealthy writes record the startup anchor
  `StartupInterval=30` → TTL 60 while still backing off. The recorded interval legitimately drops to 10 once
  the loop switches to steady-state refresh — Phase-60 D-12: "each entry records its active interval.")

### Self-watchdog (`/health/live`) semantics
- **D-03:** **Staleness-only — no verdict change.** The self-watchdog (`LivenessWatchdogHealthCheck`) is a
  **liveness** probe: it answers *"is the loop alive / should K8s restart me?"*, NOT *"is the replica
  healthy?"*. A refreshed-but-unhealthy replica therefore reports **Healthy "live"**, with `configSchema=Fail`
  surfaced in the response **body** via the PROBE-02 per-check summary data (the watchdog already serializes
  `current.Summary.{Input,Output,Config}Schema`). Flipping the verdict itself to Unhealthy **IS** the
  false-restart bug (conflating "unhealthy" with "should restart").
  - **Expected outcome:** `LivenessWatchdogHealthCheck.cs` needs **NO code change**. The fix is purely
    "keep the timestamp fresh"; the watchdog already does the right thing once `Current` stops going stale.
    (If planning discovers a change is needed, that is a signal to re-examine — the design intent is no-change.)

### Phase scope / re-proof handback
- **D-04:** **62.1 = product fix + hermetic test coverage only.** The live TEST-01b/TEST-01c re-proof (the
  ~50min Phase-62 close-gate run with real container lifecycle) is **handed back to a Phase-62 close-gate
  re-run**, not owned by 62.1. Keeps 62.1 tightly scoped to the code change + the half of validation the
  xUnit harness CAN do.

### Cross-cutting invariants (carried, not re-litigated)
- **D-05 (gate admit rule unchanged):** The orchestration-start gate (`ProcessorLivenessValidator`) must
  continue to admit **only Healthy+fresh** replicas. An unhealthy-but-fresh replica stays **BLOCKED** — but
  now **observably Unhealthy** (status=Unhealthy, key present) rather than absent. The functional block was
  already correct; this phase only changes observability. Do not touch the validator's admit logic.
- **D-06 (shutdown discipline):** The new refresh loop honors `stoppingToken` exactly like the heartbeat:
  `await Task.Delay(period, clock, stoppingToken)` wrapped to return cleanly on `OperationCanceledException`.
  A Redis fault on a refresh write is log-and-continue (the shared `ProcessorLivenessWriter` already does
  this) — a dead Redis must never crash the host or stop the loop.

### Claude's Discretion
- Exact loop placement/shape inside `ProcessorStartupOrchestrator.ExecuteAsync` (inline `while` after the
  clash block vs a small private `RefreshUnhealthyUntilShutdownAsync` helper) — as long as it reuses the
  existing `WriteUnhealthyAsync(configOutcomeOverride: SchemaOutcome.Fail)` + shared writer disciplines and
  records `interval=10` (D-02).
- Whether to reuse the period via a constructed `TimeSpan.FromSeconds(options.Value.IntervalSeconds)` or a
  cached field — mechanical.
- Exact hermetic test shape proving the refresh (drive the orchestrator's clash path on a
  `FakeTimeProvider`, advance N intervals, assert the per-instance key is re-SET with a fresh timestamp +
  reset TTL and `status=Unhealthy` each interval, and L1 `Current.Timestamp` advances). Follow the existing
  orchestrator/liveness test harness patterns.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`ProcessorStartupOrchestrator.WriteUnhealthyAsync(configOutcomeOverride)`** — already builds + writes the
  clash entry via the shared writer; the refresh loop just calls it on a timer (it reads context state on the
  orchestrator's own thread — WR-03 safe).
- **`ProcessorLivenessHeartbeat.ExecuteAsync` loop shape** — the exact cancellation-safe
  `while (!stoppingToken.IsCancellationRequested) { ...; await Task.Delay(period, clock, stoppingToken) }`
  pattern the refresh loop mirrors (including the `OperationCanceledException → return` catch).
- **`ProcessorLivenessWriter.WriteAsync`** — re-SETs the per-instance key (resets TTL), idempotent SADD,
  unconditional `_l1.Update(entry)` (so L1 `Current.Timestamp` advances each refresh), log-and-continue.

### Established Patterns
- **One writer, clean handoff at `MarkHealthy`** (Phase 60 D-01): orchestrator owns the `unhealthy` lifetime
  (startup + terminal clash), heartbeat owns `healthy` post-`MarkHealthy`. Option b preserves this — a clashed
  replica simply never reaches `MarkHealthy`, so the heartbeat never starts and there is no two-writer race.
- **Each entry records its active interval** (D-12) — the refresh loop legitimately records `interval=10`
  (steady-state) where the preceding Loop-B writes recorded `30` (backing off).

### Integration Points
- The refreshed L2 key is read by the orchestration-start gate (`ProcessorLivenessValidator`, blocks on
  status) and the index `SMEMBERS`; the refreshed L1 record is read by `LivenessWatchdogHealthCheck`
  (staleness only). Both must keep working unchanged — the fix only stops the staleness/absence.

</code_context>

<specifics>
## Specific Ideas

- The confirmed governing principle: **timestamp refresh is decoupled from health and gated only on
  "identity resolved + alive."** Every state that has a `processorId` keeps a fresh timestamp; the only
  inherent exception is the pre-identity window (no key exists yet — Phase 60 D-02).
- The fix is deliberately minimal/surgical: a ~10-line loop replacing one `return;`, no new types, heartbeat
  and watchdog untouched.
- Verify-after-fix (from the gap doc): on a fresh `docker restart sk-processor-badconfig`, the per-instance
  key `skp:proc:{id}:{instanceId}` stays present with `status=Unhealthy` and a TTL that **resets each
  interval** (refreshed indefinitely while alive), and `/health/live` reflects the real (unhealthy) status in
  its body **without** a false "loop stale." (Live verification is the Phase-62 re-proof — D-04.)

</specifics>
