<objective>
KHARD-01: bound the OUTER recover→reinject cycle with a configurable per-`H` attempt cap, landing
inside the single `KeeperRecoveryHandler` body extracted in Plan 01. A persistent (non-transient)
fault that the probe keeps "recovering" but the receiver keeps re-faulting must converge to ONE
park, not flood the stack (~67 cyc/s/replica). This is NOT `ProbeOptions.MaxAttempts` (the inner
probe-loop count) — it is a NEW cross-message per-`H` Redis counter checked in the Recovered branch.

Purpose: eliminate the unbounded reinject self-DoS (the MEMORY-tracked landmine).
Output: a `RecoverAttemptCap` option, a `skp:keeper:attempts:{H}` counter key factory, the cap
check in the shared body, a new `recover_cap` reason tag, a FakeRedis counter extension (Wave-0),
and a hermetic cap test proving exactly-cap-then-one-park.
</objective>

<decisions>
- D-A1 (single-winner park): gate the park on the atomic crossing increment `n == cap + 1` ONLY —
  mirrors the existing `flag[H]` first-writer-wins dedup (`EntryStepDispatchConsumer.cs:223-225`).
  `n <= cap` → reinject as today; `n == cap+1` → park once + DEL the key; `n > cap+1` → already
  parked, do nothing. Atomic INCR makes this race-free across the 2 keeper replicas.
- D-A2 (config): add `RecoverAttemptCap` to the existing `ProbeOptions`, default `3`, bound from the
  existing `"Probe"` section (no new Configure<T> line). Rationale: reuses the exact existing convention;
  matches the `Retry.Limit=3` budget; small enough to converge a persistent fault fast.
- D-A3 (TTL): counter key TTL = 300s (matches the flag/data window). Set TTL only on first create
  WITHOUT clobbering it on later INCRs (the keepTtl landmine — MEMORY). DEL on park. TTL-bounded so a
  missed DEL self-cleans → close-gate redis-scan triple-SHA stays net-zero.
- D-A4 (reason tag): add `KeeperMetricTags.ReasonRecoverCap = "recover_cap"`, distinct from
  `probe_exhausted`, wired into the cap-park DlqPushed tag so operators distinguish flood-cap from
  L2-never-returned.
- Test-safety (MEMORY landmine): the cap is proven HERMETICALLY only (in-memory harness + FakeRedis,
  no live keeper). A RealStack cap test is FORBIDDEN — it would flood the live stack.
</decisions>
