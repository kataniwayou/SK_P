RESEARCH.md "DLQ-1 Consolidation Mechanism" is the authoritative spec for this task. Key facts (version-correct, MT 8.5.5):
- On retry-exhaustion MT MOVES/republishes to per-`{queue}_error` (it does NOT nack-to-DLX). A naive DLX catches nothing on exhaustion (Pitfall 4). REQUIREMENTS.md DLQ-02 "confirmed in Phase-33 spike" is INACCURATE (D-07) — this wiring is UNBUILT and novel.
- Mechanism A (RECOMMENDED): `AddConfigureEndpointsCallback((ctx, name, cfg) => { cfg.UseMessageRetry(r => r.Immediate(Limit)); cfg.ConfigureError(e => { e.UseFilter(new GenerateFaultFilter()); e.UseFilter(new ConsolidatedErrorTransportFilter(/* → skp-dlq-1 */)); }); })`. The custom `IFilter<ExceptionReceiveContext>` moves the faulted message to the fixed `skp-dlq-1` endpoint, mirroring the default `ErrorTransportFilter` but with a fixed destination. GenerateFaultFilter MUST stay (Keeper's whole model rides the Fault<T> stream).
- Declare `skp-dlq-1` once with `x-message-ttl` = 7 days (ms as int/long): a no-consumer `ReceiveEndpoint("skp-dlq-1", e => e.SetQueueArgument("x-message-ttl", (int)TimeSpan.FromDays(7).TotalMilliseconds))` in the bus factory (the configureBus seam) OR via send-topology.
- Mechanism B (FALLBACK, only if A proves infeasible): per-`{queue}_error` TTL + chained DLX into skp-dlq-1. Weaker fit for D-05/DLQ-04 (delayed, not immediate consolidation). If you fall to B, flag the DLQ-03 immediacy gap to the user via a checkpoint.
- Assumptions A1/A5 (HIGH/MEDIUM): the exact `IFilter<ExceptionReceiveContext>` API + the default filter type names in 8.5.5 need a SOURCE READ before committing (the BLOCKING step below).
