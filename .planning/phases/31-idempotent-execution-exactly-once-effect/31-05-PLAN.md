<objective>
Replace the 4 hard-coded `Immediate(3)` sites with a configurable retry budget bound per process from the `Retry` appsettings section via `IOptions<RetryOptions>` (D-10). Three sites are structurally-identical orchestrator `ConsumerDefinition`s; the 4th is the BaseProcessor.Core inline `ConnectReceiveEndpoint` bind in `ProcessorStartupOrchestrator` (the landmine — it needs `IOptions<RetryOptions>` on the hosted-service ctor, not a definition).

Purpose: a single source of truth for the retry `Limit` so Phase 32's final-attempt check (`GetRetryAttempt() == Limit`) cannot desync from `UseMessageRetry`. The shape is shared (Plan 01's `RetryOptions`); the bound values are per-process.

Note (wave placement): this plan is wave 2 (depends_on [31-01] only — it needs only the shared `RetryOptions` shape from Plan 01). It runs in parallel with the contract ripple (31-02) because it touches a disjoint file set — it only adds `Configure<RetryOptions>()` bindings + appsettings `Retry` sections + retry-site `IOptions` injection, all independent of the `Guid->string` EntryId type change. File-overlap with 31-02 is zero (verified). The end-of-wave-2 `dotnet build SK_P.sln` must still compile (these changes do not touch any EntryId-typed surface).

Output: 4 retry sites bound from config + per-process registration + appsettings `Retry` sections + `RetryOptionsBindFacts` (req-7 retry half).
</objective>
