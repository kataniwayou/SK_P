# Phase 32.1: Dead-Letter on Exhaustion (Breaker Reverted) — Specification

**Created:** 2026-06-05
**Supersedes:** Phase 32 (Cancelled Circuit-Breaker) — the runtime breaker is removed
**Ambiguity score:** 0.05 (gate: ≤ 0.20)
**Requirements:** 6 locked

## Goal

On retry-budget exhaustion in either process — processor `EntryStepDispatchConsumer` or orchestrator `ResultConsumer`/Start/Stop — the message dead-letters to `_error` via the existing `UseMessageRetry(Immediate(RetryOptions.Limit))`, with **NO breaker**: no L2 cancellation marker, no in-flight check-and-drop, no `Fault`-driven Quartz unschedule. Stopping the scheduled job on exhaustion is explicitly **out of scope** (user decision, 2026-06-05). The Phase-31 `flag[H]` idempotency gates and their dedup counters are retained unchanged.

## Background

Phase 32 built a two-level "cancelled circuit breaker" — an L2 `cancelled[workflowId]` marker (in-flight stop) plus a `Fault<EntryStepDispatch>` fan-out consumer that unschedules the Quartz job (future-fire stop). After review the breaker was judged not worth its surface area: the future-fire stop was best-effort (`32-REVIEW.md` WR-01 — a persistently-faulting schedule owner produces an unconsumed `Fault<Fault<…>>`), the marker required no-TTL keys plus manual operator resume, and the "clean stop" degraded to plain `_error` precisely under the infra-down conditions that trip it.

The decision is to **drop the breaker entirely** and accept plain dead-lettering on exhaustion (the pre-Phase-32 behavior). Both receive endpoints already carry `UseMessageRetry(Immediate(RetryOptions.Limit))` → `_error` (processor dispatch-endpoint bind at `ProcessorStartupOrchestrator.cs:154`; orchestrator `ResultConsumerDefinition` / `StartOrchestrationConsumerDefinition` / `StopOrchestrationConsumerDefinition`). So this phase is a **net removal** of Phase-32's runtime additions — no new exhaustion mechanism is built.

**Out of scope:** stopping/unscheduling the Quartz job on exhaustion (off the table). **Unchanged:** `StepEntryCondition.PreviousCancelled (3)` (kept — 32-03 was cancelled), `StepOutcome.Cancelled (3)` (live for token-cancellation EXEC-08), the Phase-31 dedup counters and `flag[H]` gates, the configurable retry `Limit`.

## Requirements

### req-1 — Remove the processor breaker
- **Current:** `EntryStepDispatchConsumer` has `catch (Exception) when (ctx.GetRetryAttempt() == retryOptions.Value.Limit)` that sets the no-TTL marker, increments `WorkflowCancelled`, WARN-logs, and re-throws.
- **Target:** the catch is removed; an infra fault propagates unmodified to the endpoint-bound `UseMessageRetry(Immediate(Limit))` and dead-letters to `{processorId}_error`. The `IOptions<RetryOptions>` ctor parameter (added only for the breaker check) is removed.
- **Acceptance:** `EntryStepDispatchConsumer` references no `L2ProjectionKeys.Cancelled`, `WorkflowCancelled`, or `GetRetryAttempt`; an infra-faulting dispatch through `Limit` attempts ends in `_error` with no L2 marker written.

### req-2 — Remove both check-and-drop gates
- **Current:** `EntryStepDispatchConsumer` and `ResultConsumer` each read `L2ProjectionKeys.Cancelled(workflowId)` and ack-and-discard when set.
- **Target:** both marker reads are removed; in-flight dispatches/results process normally regardless of any cancellation state.
- **Acceptance:** neither consumer references `L2ProjectionKeys.Cancelled`; the Phase-31 `flag[H] == "Ack"` dedup gate (and its dedup counter increment) remains byte-intact in both.

### req-3 — Remove the Fault fan-out consumer
- **Current:** `FaultUnscheduleConsumer` + `FaultUnscheduleConsumerDefinition` consume `Fault<EntryStepDispatch>` on a per-replica `InstanceId`+`Temporary` endpoint wired in `Program.cs`.
- **Target:** the consumer, its definition, and its `Program.cs` endpoint registration are deleted. MassTransit still auto-publishes `Fault<EntryStepDispatch>` on exhaustion, but with no binding it is discarded at the exchange (harmless).
- **Acceptance:** no `FaultUnscheduleConsumer*` files exist; `Program.cs` has no fault endpoint; the orchestrator builds and starts with Start/Stop/Result endpoints only.

### req-4 — Remove marker + trip metric; keep dedup metrics
- **Current:** `L2ProjectionKeys.Cancelled` + `CancelledMarkerValue`, and `ProcessorMetrics.WorkflowCancelled`, exist alongside `ProcessorMetrics.DispatchDeduped` / `OrchestratorMetrics.ResultDeduped`.
- **Target:** remove `L2ProjectionKeys.Cancelled`, `CancelledMarkerValue`, and `ProcessorMetrics.WorkflowCancelled`. **Keep** `DispatchDeduped` and `ResultDeduped` and their increment sites at the `flag[H]` gates.
- **Acceptance:** grep finds no `Cancelled(` marker builder, no `CancelledMarkerValue`, no `workflow_cancelled` instrument; `processor_dispatch_deduped` and `orchestrator_result_deduped` still construct and increment.

### req-5 — Remove breaker tests + close gate; prove green + net-zero
- **Current:** Phase 32 added `BreakerTriggerFacts`, `CancelledMarkerFacts`, `CheckAndDropFacts`, `ResultCheckAndDropFacts`, `FaultUnscheduleFacts`, `FaultIdempotencyFacts`, `CancelledMarkerKeyFacts`, `BreakerMetricsFacts`, `RetryAttemptNumberingFacts`, `FaultConsumerBindingFacts`, `CancelledCircuitBreakerE2ETests`, and `scripts/phase-32-close.ps1`.
- **Target:** delete the breaker-specific tests + the E2E + `phase-32-close.ps1`. Preserve any dedup-counter coverage for the kept `DispatchDeduped`/`ResultDeduped` (re-home into an existing facts class if `BreakerMetricsFacts` is removed wholesale). Add `scripts/phase-32.1-close.ps1` (clone of `phase-31-close.ps1`) — 3×GREEN + triple-SHA; **no** `skp:cancelled:*` scan-clean needed (no such keys are written anymore).
- **Acceptance:** hermetic suite green with the breaker tests removed and zero dangling references; `phase-32.1-close.ps1` ParseFile-clean, BOM-free, FLUSHDB-grep == 0; live `GATE_EXIT=0` is the operator gate.

### req-6 — Enum + token-cancellation unchanged
- **Current/Target:** `StepEntryCondition.PreviousCancelled (3)` stays (32-03 cancelled); `StepOutcome.Cancelled (3)` stays live for the token-cancellation EXEC-08 outcome.
- **Acceptance:** no change to `StepEntryCondition.cs` / `StepOutcome.cs` from this phase.

## Notes
